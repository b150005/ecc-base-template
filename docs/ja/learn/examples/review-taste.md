## 深刻度ラダー  [JUNIOR]

### 第一原理からの説明  [JUNIOR]

深刻度ラベルのないコードレビューの指摘はノイズです。レビュワーは問題を見る; 作者はコメントを見る。共通のスケールがなければ、作者は「これはデータ損失を引き起こす」と「私なら別の名前をつけていた」を区別できません。どちらもレビューコメントとして現れます。一方はマージのブロッカーです。もう一方はオプションの改善です。

深刻度ラダーはすべての指摘をティアに割り当てます。ティアは、PR をマージする前に何が起きなければならないか、機能がリリースされる前に何が起きなければならないか、何を作者の判断に委ねるかを決定します。Meridian は 4 つのティアを使用しています:

| 深刻度 | 意味 | 必要なアクション |
|----------|---------------|-----------------|
| **CRITICAL** | セキュリティ脆弱性、データ損失リスク、またはクラッシュ | マージ前に修正必須 |
| **HIGH** | バグまたは重大な品質問題 | マージ前に修正すべき |
| **MEDIUM** | メンテナビリティの問題、クリティカルなパスのテスト欠如 | 修正を検討 |
| **LOW** | スタイルやマイナーな提案 | オプション |

レベルはコンテキストで柔らかくなりません。1 行の PR での CRITICAL 指摘は引き続き CRITICAL です。シニアエンジニアが書いたコードへの CRITICAL 指摘は引き続き CRITICAL です。ラベルは作者がフィードバックを受け入れる可能性ではなく、問題の結果を表します。明示的なルールについては [preamble §4](../.././../learn/preamble.md) を参照してください: 深刻度ラベルをレベルで柔らかくしてはなりません。

### 慣用的な応用  [MID]

Meridian はラダーを一貫して適用しています。以下の説明的なシナリオは、各レベルが実務でどのように適用されるかを示しています。

**CRITICAL シナリオ — 認可チェックの欠如。**

PR が `PATCH /v1/tasks/:task_id` エンドポイントを追加しており、ハンドラがワークスペースメンバーシップを確認せずに `svc.UpdateTask` を直接呼び出しています。どの認証済みユーザーでも、どのワークスペースに属していても、システム内の任意のタスクを更新できます。

レビュワーはこの指摘をフラグします:

```
CRITICAL: handler/task.go UpdateTask — svc.UpdateTask の前に認可チェックなし。
認証済みの呼び出し側は任意のタスクを更新できる。サービスレイヤーは更新を適用する前に
callerID が task.WorkspaceID のメンバーであることを確認しなければならない。
修正: service/task.go:UpdateTask に s.workspaces.IsMember(ctx, task.WorkspaceID, callerID) チェックを追加し、
false の場合は domain.AuthorizationError を返す。
```

これは CRITICAL です。なぜなら、すべての Meridian の顧客のタスクがすべての顧客に公開されるためです。PR はマージできません。ワークスペースメンバーシップチェックパターンについては [security-mindset → Authorization Patterns](./security-mindset.md#authorization-patterns) を参照してください。

**HIGH シナリオ — リポジトリメソッドでの誤ったエラー伝播。**

リポジトリメソッドが `translatePostgresError` を通じて変換する代わりに生の `*pq.Error` を返しています。ハンドラの `writeError` は `*pq.Error` を認識せず HTTP 500 を返します。呼び出し側は、処理できる制約違反と真のサーバーエラーを区別できません。

```
HIGH: repository/task.go Create — ユニーク制約違反で生の *pq.Error を返す。
ハンドラはこれを 409 に分類できない; 呼び出し側は意味のあるエラーコードではなく 500 を受け取る。
return の前に translatePostgresError(err) でラップすること。
```

これは HIGH です。なぜなら、リポジトリとサービスレイヤー間のコントラクトが違反されているバグだからですが、データを公開したりプロセスをクラッシュさせたりはしません。PR はマージ前に修正すべきです。Meridian が強制する変換パターンについては [error-handling → Boundary Translation](./error-handling.md#boundary-translation-from-postgres-to-domain-errors) を参照してください。

**MEDIUM シナリオ — 非自明なコードパスのテスト欠如。**

PR がタスクのアーカイブを実装しており、テストでハッピーパスをカバーしています。呼び出し側がすでにアーカイブ済みのタスクをアーカイブしようとするケースは本番で到達可能（2 人のユーザー間のレースコンディション）ですがテストされていません。

```
MEDIUM: service/task_test.go — アーカイブの冪等性（すでにアーカイブ済みのタスクをアーカイブする）のテストなし。
サービスパスは到達可能。サービスが domain.ErrConflict を返すか冪等であるかを確認する
テストを追加することを検討してください。どちらが製品上の決定かによります。
```

これは MEDIUM であり HIGH ではありません。現在のコードが積極的に誤動作しているわけではないからです — ブランチは存在してエラーを返します — しかし動作が未確認です。チームはマージしてフォローアップで対処する場合があります。テストされていないパスが HIGH にエスカレートされるタイミングについては [testing-discipline → The Meridian Test Pyramid](./testing-discipline.md#the-meridian-test-pyramid) を参照してください。

**LOW シナリオ — より正確にできる命名。**

`processData` という名前の関数がユーティリティファイルに現れています。データベースからタスクのステータス数を読み取り、ダッシュボードウィジェット用にフォーマットします。

```
LOW: internal/util/process.go processData — 名前が意図を伝えていない。
提案: FormatTaskStatusCounts または BuildDashboardSummary。
```

これは LOW です: コードは正しく、精査すれば読めます。名前はスメルですが欠陥ではありません。レビュワーが提起し、作者がマージ前に名前を変えるか後続のクリーンアップで行うかを決めます。

### トレードオフと制約  [SENIOR]

一貫した深刻度ラダーのリスクはラベルのインフレーションです: レビュワーがあらゆる不完全さに HIGH を付けると、作者は HIGH と MEDIUM を区別しなくなりシグナルが劣化します。Meridian の実践: 指摘が MEDIUM か HIGH かのどちらかである場合、コードをリリースした結果がユーザーに見えるバグになるか文書化されたコントラクトの違反になるのでなければ MEDIUM を選択します。見逃した HIGH は誤って MEDIUM に格下げするよりもコストが高くなります。

逆方向のリスクは社会的プレッシャーによるエスカレーション不足です。Meridian の PR ポリシーでは、シニア度にかかわらず認可レイヤーへの変更に 2 つの承認を必要とし、セキュリティに敏感なコードでラベルを柔らかくする誘惑を減らしています。

## 防御的プログラミングのアンチパターン: キャッチオール回復  [MID]

### 第一原理からの説明  [JUNIOR]

防御的プログラミングは、予期しない条件を予測するコードを書く実践です。Go では、これはしばしば `recover()` 呼び出しで大きなコードセクションをラップして `panic` がプロセスをクラッシュさせないようにする形をとります。意図はシステムをより耐障害性のあるものにすることです。不注意に適用されると、効果は逆になります。

あらゆる panic をキャッチして静かに汎用エラーを返す `recover()` 呼び出しは、プログラミングの不変条件が違反されたという証拠を隠します。プロセスは作者が不可能と宣言した状態で実行し続けます。システムは正常に見えます; 基礎となるデータは破損しているかもしれません。panic はシグナルでした。その状態をログに記録して影響を受けた操作を中断することなく回復することはシグナルを破棄します。

### 慣用的な応用  [MID]

Meridian はこのパターンを一括タスクアサインメントを導入した機能で経験しました。サービスメソッドはユーザー ID のリストをループし、各ユーザーに対してリポジトリの `AddAssignee` メソッドを呼び出し、ループ全体を遅延 `recover` でラップしていました:

```go
// service/task.go — アンチパターン: キャッチオール recover
func (s *TaskService) BulkAssign(ctx context.Context, taskID uuid.UUID, userIDs []uuid.UUID) error {
    defer func() {
        if r := recover(); r != nil {
            log.Warn("recovered panic in BulkAssign", "recover", r)
        }
    }()

    for _, uid := range userIDs {
        if err := s.tasks.AddAssignee(ctx, taskID, uid); err != nil {
            log.Warn("failed to add assignee", "user_id", uid, "err", err)
            // continue — 1 ユーザーのためにオペレーション全体を失敗させない
        }
    }
    return nil
}
```

遅延 `recover` は、このパスでの以前の panic がサーバーをクラッシュさせたため追加されました。正しい対応は panic の原因を見つけて修正することでした。代わりに、チームはラップして先に進みました。2 週間後、データ整合性チェックにより約 400 件のタスクアサインメントが静かにスキップされていたことが明らかになりました — `AddAssignee` は冪等性キーロジックのヌルポインタで panic しており、`recover` がそれをキャッチして nil を返し、呼び出し側はアサインメントが成功したと仮定していました。Meridian の請求コードはアサインメントレコードに基づいてシートごとに請求します; 400 件の静かにスキップされたアサインメントは、影響を受けたワークスペースで誤ったシート数を意味していました。

レビュワーはパターンをキャッチしました:

```
CRITICAL: service/task.go BulkAssign — 遅延 recover() が panic を飲み込み
nil を返し、エラーを成功としてマスクしている。呼び出し側は
「すべてのアサインメントが成功した」と「アサインメントが静かに失敗した」を区別できない。
panic がプログラミングエラーを示す場合、表面化させなければならず、飲み込んではならない。
修正: recover() を削除する。panic がこのゴルーチンをクラッシュさせる。panic の根本原因
（冪等性キーロジックのヌルポインタ）は別途修正しなければならない。運用上のエラーには
panic の回復ではなく明示的なエラーリターンを使うこと。
```

この指摘は CRITICAL です。なぜなら panic から回復して `nil` を返すことで、操作が部分的に失敗しているにもかかわらず成功を示したからです。呼び出し側のコントラクト（「nil はすべてのアサインメントが記録されたことを意味する」）が静かに違反されました。

### トレードオフと制約  [SENIOR]

Meridian で `recover` の正当な使用は 1 つあります: Gin の組み込みリカバリミドルウェアが各 HTTP ハンドラの呼び出しをラップして panic を HTTP 500 レスポンスに変換します。これは最上位レイヤーで動作し、変換前に panic をログに記録するため許容されます。これはセーフティネットであり、設計戦略ではありません。ログに記録された panic はモニタリングに表示されてアラートをトリガーします。セーフティネットの `recover` と飲み込む `recover` の違いは、エラー情報に何が起きるかです: セーフティネットの `recover` はそれを保存して表面化させ、飲み込む `recover` はそれを破棄します。

Go サービスで panic が適切な場合のポリシー全体については [error-handling → Panic Usage Policy](./error-handling.md#panic-usage-policy) を参照してください。

## 命名スメル  [JUNIOR]

### 第一原理からの説明  [JUNIOR]

関数や変数の名前はクレームです。`isTaskArchived` はタスクがアーカイブされているかどうかを示すブーリアンを返すと主張します。`processData` は何らかのデータを処理すると主張します。最初のクレームは検証できるほど具体的です: 読者は戻り値の型と実装を確認できます。2 番目のクレームはどんな実装でも満たすほど広いです。

名前が検証不能なほど広い場合、読者はコードが何をするかを理解するために実装を読まざるを得ません。名前はその役割を果たせていません。レビュワーは特定の名前のパターンをスメルとして扱うことを学びます — バグではなく、コードを二度見する必要があるシグナルです。

### 慣用的な応用  [MID]

Meridian のレビュー実践はこれらの命名パターンをフラグします:

**汎用的な関数名。** `processData`、`handleRequest`、`doWork`、`run`、`execute`、`manage`。これらの名前は危険信号です。レビューでの問いは「何と呼ぶべきか?」ではなく「この関数は実際に何をするか?」です。その答えが正しい名前です。

PR レビューのシナリオ:

```go
// internal/reporting/report.go — レビューでフラグされた
func processData(tasks []domain.Task) map[string]int {
    result := make(map[string]int)
    for _, t := range tasks {
        result[string(t.Status)]++
    }
    return result
}
```

レビューコメント:

```
LOW: reporting/report.go processData — "process" は何も伝えない。
この関数はステータスごとにタスクをカウントする。CountByStatus または GroupTasksByStatus に名前を変えること。
map[string]int を受け取る呼び出し側は statusCounts := CountByStatus(tasks) としてはるかに読みやすくなる。
```

**述語プレフィックスのないブーリアン名。** `flag`、`status`、`check`、`value`、`result` と名付けられたブーリアンは読者に何も伝えません。Meridian の規約: ブーリアンは `is`、`has`、`should`、または `can` のプレフィックスを使用します。`isArchived`、`hasPendingAssignee`、`shouldNotify` — これらは命題として読めます。`flag` は何も読めません。

```go
// service/task.go — レビューでフラグされた
var flag bool
for _, assignee := range assignees {
    if assignee.ID == callerID {
        flag = true
    }
}
if !flag {
    return &domain.AuthorizationError{...}
}
```

レビューコメント:

```
MEDIUM: service/task.go — 'flag' という名前のブーリアンは何がチェックされているかを伝えない。
isCallerAssigned または callerIsAssignee に名前を変えること。ループ条件と
認可チェックは変数の代入をトレースする必要なく正しく読める。
```

### トレードオフと制約  [SENIOR]

命名レビューコメントは、根拠なしに LOW 指摘として残す場合、レビュワーの注意予算で最も ROI が低くなります。根拠なしに「名前を変えて」を受け取った作者は指摘を却下する傾向があります。Meridian の規約では、命名コメントは良い名前なしで読者が強いられることを述べます: 「呼び出し側は戻り値を理解するために本体を読まなければならない。」これはコメントを個人的な好みからユーザーエクスペリエンスの結果として再フレーミングします。この区別が、作者がコメントを主観的な好みとして扱うか、本物の読みやすさの問題として扱うかを決定します。

命名規約に基づくこれらのレビュー判断については [implementation-patterns → Naming Conventions](./implementation-patterns.md#naming-conventions) を参照してください。

## 呼び出し側が 1 つの抽象化  [MID]

### 第一原理からの説明  [JUNIOR]

抽象化は、繰り返しまたは関連するロジックを名前付きの境界 — 関数、型、インターフェース — の背後にグループ化して、呼び出し側が詳細を知る必要がないようにする実践です。適切なタイミングで使用すると、抽象化は重複を減らし意図を明確にします。早すぎると、複雑さを減らすことなく間接層を追加します。

時期尚早な抽象化を示すスメル: コードベースで呼び出し側が正確に 1 つである関数、型、またはインターフェース。単一の呼び出し側は抽象化が正しい境界を捉えているかどうかを明らかにできません。それは到着しない 2 番目の呼び出し側を予期して書かれたかもしれません。抽象化はすべての読者に間接層を課しながら、それを正当化する恩恵（再利用、置換可能性、テスタビリティ）を何も支払いません。

### 慣用的な応用  [MID]

Meridian はリファクタリング PR でこれを経験しました。開発者がタスクの作成とタスクの更新の両方が `validateTaskTitle` を呼び出していることに気づきました。彼らは `Validate(title string) error` メソッドを持つ `TitleValidator` インターフェースと具体的な `DefaultTitleValidator` 構造体を抽出しました。インターフェースは正確に 1 つの実装と 2 つの呼び出し側を持っていました。

レビュワーのコメント:

```
MEDIUM: service/validation.go TitleValidator — このインターフェースは 1 つの実装と
2 つの呼び出し側を持ち、どちらも service/task.go の内部にある。インターフェースは
間接層（型定義、構造体、コンストラクタ、メソッド）を追加するが、モックを通じた
テスト（どちらの呼び出し側も直接テストされている）も、置換（代替実装は予見できない）も
可能にしない。

2 つの呼び出しサイトはシンプルな関数を共有できる:

  func validateTaskTitle(title string) error { ... }

異なる 2 番目の実装が現れる場合 — たとえば、異なる長さ制限を持つワークスペース固有の
タイトルバリデータ — インターフェースが正しいツールです。その強制する事態に、
それ以前ではなく導入してください。
```

ガイダンスは「インターフェースを決して使うな」ではありません。「インターフェースが何かを可能にする — テストダブル、2 番目の実装、外部呼び出し側の安定した API — をシンプルな関数ができない場合にインターフェースを導入する」です。Meridian がインターフェースを使う場所については [architecture → Hexagonal Split](./architecture.md#hexagonal-split) を参照してください: ハンドラ、サービス、リポジトリ間のレイヤー境界です。これらのインターフェースは複数の実装（本番とテストダブル）を持ちます。`domain.TaskRepository` インターフェースは時期尚早ではありません; それは重要な役割を果たしています。

Meridian が適用する決定ルール: 抽象化を導入する前に、2 番目の呼び出し側または 2 番目の実装に名前をつけてください。どちらに名前をつけることも投機的な将来のシナリオを必要とする場合、抽象化は時期尚早です。

### トレードオフと制約  [SENIOR]

1 つの呼び出し側ヒューリスティックには本物の例外があります: テスタビリティのための抽象化。データベースに書き込んだり外部 API を呼び出したりする関数は、インジェクションのための抽象化なしにユニットテストできません。1 つの本番実装と 1 つのテストダブルを持つインターフェースは時期尚早ではありません — 2 番目の「実装」はモックです。強制する事態はプロダクションレイヤーではなくテストレイヤーに存在します。

Meridian の code-reviewer は 1 つの呼び出し側の抽象化に 1 つの問いでフラグします: 「これはテスタビリティのために必要か?」もしそうなら、指摘は適用されません。そうでなければ MEDIUM として残ります。

## 2000 行のリファクタリング PR  [SENIOR]

### 第一原理からの説明  [JUNIOR]

リファクタリング PR は動作を変えることなくコードを再構成します。新機能なし。バグ修正なし。再構造化のみ。リファクタリング PR の暗黙のコントラクトは、すべてのテストが前後でパスし、挙動の変更が導入されていないことです。レビュワーはリファクタリング PR を「何も変わっていない」という理由で軽く精査する傾向があります。これは PR サイズと強く相関する間違いです。

2000 行のリファクタリング PR はリファクタリングではありません; リファクタリングのラベルを付けた書き直しです。そのサイズでは、レビュワーは変更された各行が以前何をしていて今何をするかの完全なコンテキストを保持できません。挙動の変更は導入しやすく見落としやすいです。テストはリファクタリングの一部として書き直された場合でもパスする場合があります。

### 慣用的な応用  [MID]

Meridian の PR ポリシーは: 「リファクタリング」とラベル付けされた PR は Go コードの最大 400 ネット行（テストファイルを除く）に触れることができます。この制限を超える PR は、各コミットがテストスイートで独立してグリーンになる連続コミットに分割しなければなりません。

このポリシーは、2000 行の「名前変更と再編成」PR がタスクのパーミッションチェックに挙動変更を導入した後に確立されました。元の `canArchive` メソッドはワークスペースのオーナーシップをチェックしていました。リファクタリング後、抽出された `ArchivePermission` 構造体はオーナーシップチェックを省略してタスクのステータスのみをチェックしていました。旧と新の両方の実装が既存のテストをパスしました。テストダブルがプロダクションコードとともにリファクタリングされ、モックの動作がそのプロセスで簡略化されたためです。

バグはマージから 2 日後に顧客サポートのチケットによって発見されました。タスクをアーカイブするには `active` ステータスにあるだけでよくなっており; ワークスペースメンバーシップはもはや必要ではありませんでした。どの認証済みユーザーでも任意のアクティブなタスクをアーカイブできました。

レビュワーは PR のサイズをフラグしていました:

```
HIGH: この PR は「リファクタリングのみ」とラベル付けされ 23 ファイルにわたって 2134 行に触れている。
このサイズのリファクタリングは挙動の保持のためにレビューできない。
要求: それぞれが独立してグリーンな、より小さい PR の連続に分割すること。
提案された順序:
  1. ファイルの名前変更と移動（ロジック変更なし）— 約 300 行
  2. ArchivePermission の抽出 — 約 200 行、ワークスペースメンバーシップチェックが
     保持されることを表明するテストを含めること
  3. サービスへの新しい型の配線 — 約 100 行
```

作者は指摘に対応せずにマージしました（承認者の 1 人は HIGH が解決されているかどうかを確認していませんでした）。パーミッションのバグが結果でした。Meridian のプロセスは、2 番目の承認が付与される前に HIGH 指摘が解決されるか明示的に延期されることを今では要求しています。テストスイートがその後どのように強化されたかについては [testing-discipline → The Meridian Test Pyramid](./testing-discipline.md#the-meridian-test-pyramid) を参照してください。

### トレードオフと制約  [SENIOR]

400 行の制限は、大きな名前変更が真に挙動保存的である場合に摩擦を生み出します。そのような PR を分割するには、ブランチの依存関係と中間 CI 実行の調整が必要です。Meridian はこのコストを受け入れます。認可レイヤーで挙動の変更を見落とすコストのほうが高いためです。制限は下限であり、目標ではありません。

例外: ツールによってコードベース全体に手動編集なしで適用された自動リファクタリング。`goimports -l ./...` や機械的な `sed` 名前変更の出力である 3000 行の PR はレビュー可能です。レビュワーはすべての変更行を読む代わりにツールの動作を確認できるためです。重要な基準は機械的な検証可能性です。このスケールでの人間が書いたリファクタリングはそのようにして検証できません。

## これまでの理解: すべてのスタイルの指摘がレビューコメント  [JUNIOR]

### これまでの理解 (改訂 2026-01-14)

> 元の Meridian のレビュー実践では、レビュワーが PR コメントですべてのスタイルの逸脱をフラグすることを要求していました。フォーマット、インポートの順序、変数の命名規約を含みます。意図は一貫性を強制することでした。効果は、200 行の機能追加に 30〜50 の LOW コメントが蓄積することで、ほとんどの作者は根拠を読まずに機械的に対応していました。

> 元の実践: 「深刻度にかかわらず、スタイルガイドからのすべての逸脱をレビューコメントでフラグする。」

**修正後の理解:**

リンターとフォーマッターは機械的なスタイルルールを強制するために存在し、レビュワーがそうする必要がないようにします。CI チェックがすべての PR で `gofmt`、`golangci-lint`、`goimports` を実行して違反でビルドを失敗させる場合、それらの指摘をコメントで再び提起する人間のレビュワーはリンターの仕事を複製して、作者がすでに持っている情報でレビューキューを占有しています。

Meridian はその実践を改訂しました: レビュワーの注意は、リンターがキャッチできない指摘のために確保されます — 設計の問題、技術的には正しいが誤解を招く命名、観察可能な挙動のためのテスト不足、時期尚早な抽象化、コピーペーストのロジック、深刻度をエスカレートさせるパターン。リンターがすでに強制しているスタイルの指摘は、人間のレビューコメントで繰り返しません。

実際的な結果: 200 行の PR での平均レビューコメント数は約 35 から約 8 に減少しました。残ったコメントは平均してより高いシグナルを持つようになりました: 設計レベルの観察、MEDIUM と HIGH の指摘、クロスドメインのノート（認可、エラー処理）。それらのコメントに対する作者の応答率が向上しました。

根底にある原則: レビュワーの注意は有限です。機械がすでに報告した指摘に費やすことは、人間だけが行える指摘のために使える注意を減らします。スタイルについてリンターを信頼することは低い基準ではありません — それは人間のレビュワーに対するより高い基準です。

### 関連節

- リンターが意味を検出できず構文のみを検出するため、人間のレビュワーのスコープに残る命名の指摘については [review-taste → Naming Smells](#naming-smells) を参照してください。
- 人間のレビューに属する（リンターではない）テストされていないパスのシナリオについては [testing-discipline → The Meridian Test Pyramid](./testing-discipline.md#the-meridian-test-pyramid) を参照してください。

### コーチイラストレーション（default vs. review-only）

> **説明のみ。** 以下は、このドメインのタスクに対して 2 つのコーチングスタイルがどのように異なるかを示す作業事例です。ライブエージェントのコントラクトの一部ではありません。実際のエージェントの動作は `.claude/skills/learn/coach-styles/` のスタイルファイルによって規定されます。

**シナリオ:** 学習者がタスクの一括キャンセルエンドポイントを追加する PR を提出します。サービスメソッドはすべてのエラーを `recover()` ブロックで捕捉し、警告としてログに記録します。レビュワーエージェントが差分をレビューするために呼び出されます。

**`default` スタイル** — エージェントが差分全体をレビューして完全な指摘レポートを作成します。`recover()` ブロックを CRITICAL に分類し（BulkAssign インシデントと同じパターンで、操作の失敗を成功としてマスクする）、欠けている認可チェックを CRITICAL に（認証済みの任意のユーザーが任意のタスクをキャンセルできる）、`done` という名前のブーリアンを LOW に（`isCancelled` に名前を変える）、部分的なキャンセル失敗のテスト不足を MEDIUM に分類します。レポートはすべての 4 つの指摘を深刻度、場所、提案された修正とともにリストします。エージェントはキャッチオールのアンチパターンと深刻度ラダーを説明する `## Learning:` トレーラーを追加します。

**`review-only` スタイル** — 学習者がサービスメソッドを書いて提出すると、エージェントはハンドラやテストの記述を断ります。次のように応答します:

```
## Coach: review

アクティブなコーチスタイルは review-only です。プロダクションコードの提出は利用できません。
提出されたサービスメソッドをレビューします:

CRITICAL [service/task.go:BulkCancel]: 遅延 recover() が panic 時に nil を返し、
部分的な失敗を成功としてマスクしている。recover() を削除する; 基礎となる
nil ポインタの参照解除を直接修正すること。

CRITICAL [service/task.go:BulkCancel]: タスクを変更する前にワークスペースメンバーシップチェックがない。
ループの前に s.workspaces.IsMember(ctx, ...) を追加すること。

MEDIUM [service/task_test.go]: バッチ内の 1 つのタスクがキャンセルできない場合のテストがない
（ステータスが間違っている）。期待される動作（フェイルファスト vs. 部分的成功）は
テストで文書化すべきです。

LOW [service/task.go:39]: 'done' という名前のブーリアン — isCancelled に名前を変えること。
```

`<!-- coach:review-only stop -->`

学習者が CRITICAL の指摘を修正して再提出します。エージェントが更新された差分をレビューし、認可チェックが存在することを確認し、残留する MEDIUM の指摘を報告します。
