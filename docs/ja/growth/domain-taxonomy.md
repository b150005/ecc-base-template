# ドメイン分類体系 — Developer Growth Mode ノート

このドキュメントは、Growth Mode ノートが蓄積される知識ドメイン、各ドメインに貢献するエージェント、および時間の経過とともにドメインファイルが成長していく際の構造を定義します。

Developer Growth Mode は固定的なカリキュラムを備えた機能ではありません。実際の作業のなかで生じる学びの機会を捉えるための仕組みです。ノートは日付ではなくドメイン別に整理されます。時間が経つにつれて、ドメインファイルは多くのセッションを通じて少しずつ構築された個人用のリファレンスへと育ちます。このプロジェクトではこれを「生きたリファレンス」と呼びます。

---

## 目的と設計思想

### Growth Note とは

Growth Note は、エージェントの出力に添えられる短く検証可能な説明であり、なぜその判断が下されたのかを学習者に伝えるものです。パターンに名前を与え、トレードオフを説明し、あるいは正典となるドキュメントを参照します。Growth Note は成果物そのものを書き換えることはなく、成果物に対して後から付けられる注記です。

### ノートは何になるか

個々のノートは `.claude/growth/notes/` 配下のドメインファイルに蓄積されていきます。ドメインファイルは、時系列・セッション単位で積み上がるジャーナルでも、編集者がキュレーションする Wiki でも、教育的な順序に沿って並べられたカリキュラムでもありません。**ドメイン別に整理されたリファレンス**であり、エントリは時間とともに深まり、洗練され、ときには以前の記述を置き換えます。数週間あるいは数ヶ月後に同じドメインファイルを開き直したとき、前回読んだときよりも内容が充実していると感じられる——そのような性質を目指します。

### セッションではなくドメインで整理する理由

「2026-04-22 のセッションノート」のようにセッション単位で整理されたノートは、振り返りには役立つものの一過性のものです。エラーハンドリングに取り組む開発者にとっては、「20 回のセッションログにエラーハンドリングのメモが散在している」状態よりも、「あらゆるセッションから得たエラーハンドリングの知見が 1 つのファイルにまとまり、相互リンクされている」状態のほうが圧倒的に有用です。ドメイン単位の整理は、再利用のためのものです。

## 19 のドメイン

各ドメインは、エージェントが作業のなかで知見を寄せるにつれて育っていく、独立した知識領域です。以下では各ドメインについて、ファイル名、対象範囲の定義、そして各経験レベルの開発者がそのドメインで何を学ぶのかという観点を示します。これら 19 のドメインに加え、学習者は必要に応じて `.claude/growth/notes/custom/` 配下にカスタムドメインファイル（例: `on-call-readiness`、`accessibility-audit`）を作成できます。

### 1. architecture — アーキテクチャ

**ファイル**: `architecture.md`

**対象範囲**: システムの高レベル構造、モジュール境界、レイヤリング、データフロー、統合パターン、アーキテクチャスタイル（レイヤード、イベント駆動、マイクロサービスなど）、ドメインモデリング、集約設計。実装の詳細（ループ、条件分岐、変数の命名など）は `implementation-patterns` の対象であり、本ドメインには含まれません。

**Junior の観点**: アーキテクチャスタイルの用語を学び、ある問題に対してあるスタイルが選ばれる理由を理解します。

**Mid の観点**: スタイル間のトレードオフを理解し、選ばれたスタイルが問題領域に照らして自明ではない選択である場合にそれを見抜けるようになります。

**Senior の観点**: 現在および将来の制約に照らしてアーキテクチャ上の判断を評価し、コストやリスクの変化がスタイル変更を正当化し得ることを認識します。

**時間をかけて育っていく節の例**:
- レイヤードアーキテクチャ — いつ採用すべきか、レイヤを重ねすぎることのコスト
- イベント駆動パターン — イベントソーシング、CQRS、それらが正当化される状況
- マイクロサービス — モノリスとの比較、スケールが複雑さを正当化する場面
- ドメイン駆動設計 — 集約の境界、ユビキタス言語
- アンチパターン — 選んだスタイルが問題と噛み合わなくなる場面

---

### 2. api-design — API 設計

**ファイル**: `api-design.md`

**対象範囲**: API の契約、リクエスト／レスポンス形式、バージョニング戦略、エラーコード、ページネーション、フィルタリング、レート制限、冪等性、REST の規約、gRPC のパターン、Webhook の設計。ルーティングの仕方やシリアライズ／デシリアライズといった実装の詳細は `implementation-patterns` の対象であり、本ドメインには含まれません。

**Junior の観点**: REST の規約を学び、エンドポイントがどのようにグループ化・命名されるのか、その理由を理解します。

**Mid の観点**: バージョニングのトレードオフを理解し、API を破壊的変更するのか拡張にとどめるのかを判断できるようになります。

**Senior の観点**: クライアントとの結合度、変更速度、長期的な保守コストに照らして API 設計を評価します。

**時間をかけて育っていく節の例**:
- REST のリソース階層 — フラット構造とネスト構造の比較、ネストを深くしない理由
- エラーレスポンスの形式 — このプロジェクトが採用したエンベロープ方式
- ページネーション — カーソルベースとオフセットベースの比較、使い分けの基準
- API のバージョニング — ヘッダーベースと URL パスベースの比較、サンセット期間の設計
- レート制限とクォータ — シグナルによる抑制と拒否、クライアント体験への影響

---

### 3. data-modeling — データモデリング

**ファイル**: `data-modeling.md`

**対象範囲**: エンティティ設計、関連、正規化と非正規化、集約の境界（DDD の文脈での意味）、時間軸を持つデータ、ステートマシン、レコードの不変性。SQL DDL などのスキーマ構文やデータベース固有の機能は `persistence-strategy` の対象であり、本ドメインには含まれません。

**Junior の観点**: エンティティがその形をしている理由と、集約の境界が何を意味するのかを学びます。

**Mid の観点**: 正規化と非正規化の使い分け、および時間軸を持つデータ（監査証跡、緩やかに変化する次元=SCD）のモデル化の仕方を理解します。

**Senior の観点**: 整合性、監査可能性、将来のクエリパターンの観点からデータモデルを評価します。

**時間をかけて育っていく節の例**:
- 集約の設計 — 1 つの集約に含めるべきもの、切り出すべきもの
- 時間軸を持つデータのモデリング — 監査ログ、論理削除、イベントのタイムスタンプ
- 非正規化の判断基準 — データを複製することが、そのコスト以上の価値をもたらす場面
- ステートマシン — boolean フラグの寄せ集めではなく、状態そのものをモデル化する
- 不変性の契約 — 追記専用のレコード、コピーオンライト

---

### 4. persistence-strategy — 永続化戦略

**ファイル**: `persistence-strategy.md`

**対象範囲**: データベース技術の選定、スキーマ設計、インデックス、クエリパターン、トランザクション、整合性モデル（ACID と結果整合性）、CRUD 操作の設計、リレーショナル／ドキュメント／キーバリュー／時系列の各データベース。データに関するビジネスロジックは `data-modeling` の対象であり、本ドメインには含まれません。

**Junior の観点**: 特定のデータベースが選ばれた理由、および基本的なインデックス戦略とクエリパターンを学びます。

**Mid の観点**: 整合性に関するトレードオフを理解し、結果整合性を許容できるのはどのような場面かを把握します。

**Senior の観点**: 可用性、レイテンシ、運用コストの観点から永続化に関する判断を評価します。

**時間をかけて育っていく節の例**:
- データベースの選定 — リレーショナル／ドキュメント／キャッシュの比較、このプロジェクトがその選択に至った理由
- インデックス戦略 — ある列にインデックスを張る理由、書き込みレイテンシとのトレードオフ
- クエリパターン — N+1 問題の予防、JOIN の戦略、集計
- 整合性モデル — ACID トランザクションと結果整合性
- 運用上の考慮 — バックアップ、リカバリ、スケーリング

---

### 5. error-handling — エラーハンドリング

**ファイル**: `error-handling.md`

**対象範囲**: エラーの種類、エラーの伝播、利用者に見せるエラーメッセージ、エラーのログ記録と可観測性、リカバリ戦略、防御的プログラミング。Result 型と例外のような構文レベルの話題は `ecosystem-fluency` の対象であり、本ドメインには含まれません。

**Junior の観点**: このプロジェクトにおけるエラーの分類と、それぞれがどのような場面で使われるのかを学びます。

**Mid の観点**: エラー伝播の境界を理解し、レイヤー間でエラーを変換すべき場面を把握します。

**Senior の観点**: 可観測性、利用者体験、運用上のリカバリという観点からエラー戦略を評価します。

**時間をかけて育っていく節の例**:
- エラーの分類 — 回復可能なものと致命的なもの、利用者起因のものとシステム起因のもの
- 境界をまたぐエラー — レイヤー間でエラーがどのように変換されるか
- 利用者向けのメッセージング — 何を表示し、何をログだけにとどめるか
- 可観測性のためのログ設計 — 構造化ログ、コンテキストの引き回し
- 耐障害性 — リトライロジック、サーキットブレーカー、グレースフルデグラデーション

### 6. testing-discipline — テスト規律

**ファイル**: `testing-discipline.md`

**対象範囲**: テスト戦略、テストの構造、モックと統合テストの使い分け、フィクスチャとテストデータ、テストピラミッド。アサーションライブラリなどのテスト構文に関する話題は `ecosystem-fluency` の対象であり、本ドメインには含まれません。

**Junior の観点**: AAA パターンを学び、それぞれのセクションが重要である理由を理解します。

**Mid の観点**: どのようなときにモック化するか、どのようなときに統合するかを理解し、テスト間の結合を避ける方法を身につけます。

**Senior の観点**: 速度、信頼性、保守コストの観点からテスト戦略を評価します。

**時間をかけて育っていく節の例**:
- 不変条件の階梯 — 各テストレベル（ユニット／統合／E2E）が何を証明するのか、その理由
- モックと統合の比較 — トレードオフ、それぞれが価値を発揮する場面
- フィクスチャ設計 — インラインと共有の比較、状態の結合を防ぐ方法
- テストデータ戦略 — ファクトリ、ビルダー、現実的なデータ
- テストの不安定さ（flakiness） — 根本原因と予防策

### 7. concurrency-and-async — 並行性と非同期

**ファイル**: `concurrency-and-async.md`

**対象範囲**: 並行パターン、async/await、Promise、コールバック、チャネルパターン、ロックと同期、競合状態、デッドロックの予防。言語固有のイディオムにも触れますが、主眼はパターンそのものに置きます。

**Junior の観点**: 並行実行に関する思考モデルを学び、特定のパターンが競合状態を防ぐ理由を理解します。

**Mid の観点**: 非同期の境界を理解し、ブロックすべきか待機すべきかを判断できるようになります。

**Senior の観点**: スループット、レイテンシ、リソース利用の観点から並行戦略を評価します。

**時間をかけて育っていく節の例**:
- 並行実行のモデル — スレッド、async/await、グリーンスレッド、goroutine
- 同期化のパターン — ロック、チャネル、アトミック操作
- 競合状態の回避 — 不変性、分離、同期化
- デッドロックの予防 — ロック順序、タイムアウト
- 非同期の境界 — タスクを起動するタイミングと合流（join）するタイミング

### 8. ecosystem-fluency — エコシステムへの習熟

**ファイル**: `ecosystem-fluency.md`

**対象範囲**: 言語の慣用表現、ツールチェーンの規約、フレームワークのパターン、コミュニティにおけるベストプラクティス。Go のインターフェース設計、Dart/Flutter の null 安全性、Python の dataclass と NamedTuple の使い分け、TypeScript の moduleResolution、慣用的なエラー型、そして標準ライブラリと外部パッケージのどちらを選ぶかという原則などを扱います。パッケージの選定やバージョン管理は `dependency-management` の対象であり、本ドメインには含まれません。

**Junior の観点**: 言語およびフレームワークの規約を学び、「慣用的（idiomatic）」という言葉が何を意味し、なぜそれが重要なのかを理解します。

**Mid の観点**: 自明ではない慣用表現の差異を理解し、それぞれをいつ適用すべきかを判断できるようになります。

**Senior の観点**: 慣用表現のトレードオフを評価し、意図して規約を破るべき場面を見極めます。

**時間をかけて育っていく節の例**:
- 言語の規約 — Go のインターフェース、Python の dataclass、Rust のトレイト境界、TypeScript のジェネリクス
- フレームワークのパターン — Flutter のウィジェット構成、Django の ORM イディオム、Axum のマクロ駆動ルーティング
- 慣用的なエラーハンドリング — Go の多値返却によるエラー、Rust の Result 型、Python の例外階層
- 標準ライブラリと外部パッケージ — どちらに頼るかを判断するための原則
- 標準的な命名とスタイル — enum のケース、関数の命名、構造体フィールドの規約
- 並行性の慣用表現 — goroutine とチャネル、async/await、アクターモデル

### 9. dependency-management — 依存関係管理

**ファイル**: `dependency-management.md`

**対象範囲**: パッケージの選定、バージョン制約、推移的依存関係の管理、ベンダーロックイン、アップグレード戦略。直接依存と推移的依存の両方を扱い、主にサプライチェーンに関する懸念とバージョン固定の判断に焦点を当てます。慣用表現の差異は本ドメインの範囲外です。

**Junior の観点**: ある特定のパッケージが選ばれた理由と、そのパッケージの安定性を推し量るための視点を学びます。

**Mid の観点**: 推移的依存に伴うリスクを理解し、バージョンを固定すべきか範囲指定で許容すべきかを判断できるようになります。

**Senior の観点**: 保守、セキュリティ、ベンダーリスクといった依存関係のコストと、そこから得られる利益を比較して評価します。

**時間をかけて育っていく節の例**:
- パッケージの選定 — このプロジェクトが Y ではなく X に依存している理由
- バージョン制約 — 固定と範囲指定の比較、リスクの管理方法
- 推移的依存 — 推移的依存が問題を引き起こす場面
- セキュリティアップデート — その頻度、緊急度、優先順位の付け方
- ベンダーリスク — パッケージが保守されなくなる、あるいは敵対的になるケース

### 10. implementation-patterns — 実装パターン

**ファイル**: `implementation-patterns.md`

**対象範囲**: モジュール内のコード構成、ヘルパー関数、変数の命名、制御フロー（ループ、条件分岐）、リファクタリング戦略、コードスメル。アーキテクチャやデータモデリング、構文レベルの話題は本ドメインには含まれません。

**Junior の観点**: プロジェクトで共通的に使われるパターン（ページネーション、フィルタリング、状態の更新など）を学びます。

**Mid の観点**: いつ処理を関数として抽出するか、いつインラインに残すか、そして意図が明確に伝わる命名の方法を理解します。

**Senior の観点**: 可読性、テスト容易性、保守コストの観点から実装を評価します。

**時間をかけて育っていく節の例**:
- 共通パターン — このスタックにおけるページネーション、フィルタリング、状態更新
- 早期 return とネストした条件分岐 — それぞれを使うべき場面
- 抽出の判断基準 — ヘルパー関数にすることで意図が明確になる場面
- 命名規約 — boolean、コレクション、状態変数の命名
- コードスメルの兆候 — 重複、長すぎる関数、神オブジェクト

### 11. review-taste — レビューの勘どころ

**ファイル**: `review-taste.md`

**対象範囲**: コードレビューで働かせるヒューリスティクス、良い設計の特徴、レビュー時にレビュアーが指摘しがちな問題、押し付けにならず行動に移しやすいフィードバックの伝え方。形式化しにくい「良さ」への感覚を、学びのなかで言語化・体系化していくドメインです。

**Junior の観点**: レビュアーが何に目を向けているか、なぜ特定のパターンが好まれるのかを学びます。

**Mid の観点**: それらの優先順位の背景にある考え方を理解し、異議を唱えるべき場面を見極められるようになります。

**Senior の観点**: 一貫した原則に基づくレビュー哲学を築き、そのルールから外れるべき場面を判断できるようになります。

**時間をかけて育っていく節の例**:
- CRITICAL／HIGH／MEDIUM が意味するもの — 各レベルの深刻度の基準
- 設計レビューの勘 — テストが通っていても何かおかしいと感じ取る力
- テストの深さ — テストが十分なとき、不足しているとき
- 命名の明確さ — 名前そのものがコードスメルになっている場面
- トレードオフの判断 — 完璧さよりも「十分に良い」が勝る場面

### 12. security-mindset — セキュリティマインドセット

**ファイル**: `security-mindset.md`

**対象範囲**: 典型的な脆弱性、入力バリデーション、シークレット管理、認証と認可の設計、脅威モデリング。セキュリティツールやスキャン機能は `operational-awareness` の対象であり、本ドメインには含まれません。

**Junior の観点**: このプロジェクトのスタックという文脈のなかで OWASP Top 10 を学びます。

**Mid の観点**: 脅威モデルを理解し、特定の脆弱性が検討対象となる場面を見極められるようになります。

**Senior の観点**: アーキテクチャ上のセキュリティ判断と、長期的なリスクに対する姿勢を評価します。

**時間をかけて育っていく節の例**:
- 入力バリデーション — どの地点で検証すべきか、どこまでを信頼するか
- シークレット管理 — 環境変数、シークレットボルト、ローテーション戦略
- 認証の境界 — トークンを信頼できる場面と、あらためて確認すべき場面
- 認可のパターン — ロールベース、属性ベース、ケイパビリティベース
- このスタックで起こりやすい攻撃経路 — SQL インジェクション、XSS、CSRF

### 13. performance-intuition — パフォーマンスに関する直感

**ファイル**: `performance-intuition.md`

## ノートエントリの構造

ドメインファイルは時間をかけて育っていきます。この節では、その中身がどのような形をとるのかを定義します。

### フロントマター

すべてのドメインファイルは、次のようなメタデータから始まります。

```markdown
---
domain: architecture
last-updated: 2026-04-22
contributing-agents: architect, implementer, code-reviewer
---
```

新しい節が追加されたとき、あるいは大きな改訂が加わったときには `last-updated` を更新してください。新しいエージェントが内容を追記したときには `contributing-agents` を更新してください。

### 本文の構成

本文は日付やセッションではなく、**概念（concept）** を単位として整理します。概念とは、ドメインのなかで時間を超えて通用するテーマのことです。たとえば `architecture.md` であれば、「集約の設計」「イベント駆動パターン」「レイヤリングのアンチパターン」といった概念が育っていくはずです。

それぞれの概念は、次の構成をとります。

1. **概念のタイトル**（H2。例: `## 集約の設計`）
2. **第一原理からの説明**（300〜500 語、Junior 向け） — 読者がこの概念に初めて触れるものとして書きます。明快に、具体的に、プロジェクト内の実コードを例に挙げて説明します。
3. **このプロジェクトでの実践**（200〜300 語、Mid 向け） — このプロジェクトではどう実践しているか。教科書的な説明と比べて何が自明ではないのか。
4. **トレードオフと制約**（200〜300 語、Senior 向け） — いつそれが正しい選択であり、何がそのコストなのか。代替案として何があり、なぜそれらは選ばれなかったのか。
5. **例**（必要に応じて） — コードスニペット、図（アスキーアートでも文章での記述でも可）、または `docs/en/adr/` 配下の ADR への参照。
6. **関連する概念** — 同じドメイン内、あるいは隣接ドメインにある、関連する他の概念へのリンク。
7. **改訂履歴**（概念が変化した場合のみ） — 「Prior understanding (revised YYYY-MM-DD)」というタイトルの日付付きサブ節に、以前の理解や変更の概要を記録します。

## 不変条件の階梯

**不変条件の階梯** は、テストカバレッジを考えるための思考モデルです。各テストレベルが何を *証明* するのかを明確にし、途中の段を飛ばすことができない理由を示してくれます。

### 第一原理からの説明

ユニットテストは、既知の入力に対して関数が期待される出力を返すことを証明します。実体を持つのはテスト対象の関数だけで、データベースはモック化され、外部 API はスタブに置き換えられます。ユニットテストは高速です（ミリ秒単位）。

統合テストは、2 つ以上のコンポーネントが正しく連携することを証明します。決済ゲートウェイやサードパーティ API のような、プロジェクトの管理外にある境界はモック化する一方で、データベースへの実際のクエリ、ファイル I/O、ネットワーク呼び出しは実行します。統合テストはこれよりも遅くなります（秒単位）。

E2E テストは、利用者が実際に操作したときにシステムが期待どおりに動くことを証明します。実際のブラウザ、実際のバックエンド、実際のデータベース（あるいは本番相当のレプリカ）を用います。E2E テストは最も遅く、分単位の時間を要します。

なぜこの 3 つのレベルが必要なのでしょうか。各レベルが、下のレベルでは証明できないことを証明するからです。ユニットテストだけでは、関数が実運用のスキーマ上で動くかを確かめられません。統合テストだけでは、UI が API のレスポンスを正しく解釈しているかを確かめられません。E2E テストは最後の砦ですが、実行コストが高く、頻繁には回せません。

このプロジェクトは **テストピラミッド** を採用しています。安価で高速なユニットテストを多数、より高価な統合テストをそれより少数、最も高価な E2E テストをごく少数、という構成です。このピラミッド型により、不具合は早期に検出され、高価なテストは利用者に影響する重要なシナリオだけに温存されます。

### このプロジェクトでの実践

このプロジェクトではユニットテストを Arrange-Act-Assert（AAA）パターンで構造化します。

```
Arrange: 入力とモックをすべてセットアップする
Act:     関数を呼び出す
Assert:  結果を検証する
```

Assert のブロックにロジックは置きません。条件分岐もループも書きません。アサーションが失敗したとき、テストのファイル行番号は失敗箇所を直接指し示します。どのアサーションが失敗したのかが分岐によって曖昧になることがないため、失敗原因をすぐに解釈できます。

統合テストでは、`beforeEach` フックで共有するフィクスチャではなく、テストごとにインラインで組み立てるフィクスチャを用います。共有フィクスチャは、時間が経つにつれて様々な要件を背負い込み、テスト間の結合の温床になります。テスト A とテスト B が同じ共有フィクスチャに依存していると、テスト A の要件を変えたとたんにテスト B が黙って壊れる、ということが起こります。インラインフィクスチャは多少の重複を許容しますが、この結合を防いでくれます。

### トレードオフと制約

ユニットテストは安価ですが、誤った安心感を与えることがあります。関数単体ではうまく動いても、他のコンポーネントと統合した途端に壊れることがあるからです。統合テストが不足していれば、不具合をステージングや本番環境まで持ち込むリスクがあります。一方で E2E テストが多すぎれば、CI が遅くなりフィードバックサイクルが長くなります。

このプロジェクトではユニットテストのカバレッジ 80% を目標とし、統合テストはシステム境界をまたぐコード（データベースクエリや外部 API 呼び出し）に絞り、E2E テストは重要なユーザーフローだけに限定します。こうして、確信度と CI の所要時間のバランスを取っています。

古いファイルでカバレッジが 40% しかないときは、そのファイルは移行の候補です。新機能にテストが 1 つも書かれていないときは、コードレビューにおける CRITICAL 指摘となります。

### 例

ページネーション付きのクエリ関数の場合、次のように使い分けます。

- ユニットテスト: データベースをモック化し、関数が正しい SQL を組み立てること、空の結果セットを適切に処理することを検証します。
- 統合テスト: テスト用の実データベースを使い、15 件の結果を返すクエリが最初のページで 10 件の結果と正しい `nextToken` を返すことを検証します。
- E2E テスト: 利用者が「もっと読む」をクリックしたときに、ページが明滅することなく新しいアイテムが追加されることを検証します。

それぞれのテストレベルは異なることを証明します。統合テストを省略すると、利用者が実際に操作するまで SQL の構文エラーに気づけない、といったことが起こりえます。

### 関連する節

- `error-handling` → 「Logging for Observability」: 失敗したテストは、豊富なコンテキスト付きでログされたときに初めて本領を発揮します。
- `review-taste` → 「Testing Depth」: テストカバレッジに関するレビュアーのフィードバックが建設的かどうかを見極める方法。
- `implementation-patterns` → 「Early Return vs. Nested Conditionals」: AAA テストは、アサーションをフラットに保つために早期 return を好みます。

### Prior Understanding (revised 2026-04-20)

この節の以前のバージョンは、「ユニットテスト」と「モックを使ったテスト」を混同していました。上記では両者の区別を明確にしています。ユニットテストは、テスト対象の関数を他から切り離して検証するものです。統合テストは、外部境界をモック化しつつ、実在するコンポーネント同士の相互作用を検証するものです。あるテストが「隔離されている」からといって必ずしもユニットテストとは限りません。実際のデータベースコードを実行しているのであれば、それは統合テストに分類されます。

## インターフェースを受け取り、具象型を返す（およびその裏返し）

この原則は、複数の言語にまたがって最も慣用的なパターンの 1 つを説明するものです。呼び出し側には柔軟で、実装側には明確な境界を設計する方法であり、具体的な形は言語によって違っても、原則そのものは一貫しています。

### 第一原理からの説明

関数を設計するとき、私たちは非対称な 2 つの判断を下します。呼び出し側に何を *要求* するか（入力）と、呼び出し側に何を *保証* するか（出力）です。

入力について。関数が特定の具象型（たとえば `DatabaseConnection` クラス）を要求してしまうと、呼び出し側はその型でしかその関数を使えません。互換性のあるもの（たとえばモック用の接続オブジェクト）を持っていても、型が一致しなければ渡せません。慣用的な設計は、**インターフェースを受け取る** ことです。「クエリを実行できるものなら何でもよい」という形にしておけば、呼び出し側は本物の接続、モック、実接続をラップしたクエリロガーなど、必要なメソッドを備えた任意のオブジェクトを渡せます。

出力について。インターフェース（たとえば「`database.Connection` として振る舞うもの」）を返してしまうと、呼び出し側は中身の分からない型を受け取ることになります。実際の接続オブジェクトはインターフェースが公開していないメソッドを持っているかもしれませんが、呼び出し側がそれらを使いたければ戻り値をキャスト（型アサーション）しなければならず、実行時に失敗する可能性が生まれます。慣用的な設計は、**具象型を返す** ことです。呼び出し側は実際の接続オブジェクトを受け取り、それが何であるかを知ることができます。インターフェースは *約束事の文書* であり、実際に返されるものの上限でも下限でもありません。

### 各言語での実現

- **Go**: 関数は `io.Reader`（インターフェース）を受け取り、`*os.File`（具象型）を返します。これはどの場面でも慣用的な書き方です。
- **Java / C#**: メソッドは `List<T>`（インターフェース）を受け取り、`ArrayList<T>` を返します。Java の標準ライブラリは歴史的経緯からインターフェースを返す傾向がありますが、それはこのパターンの例外です。
- **Python**: ダックタイピングにより、入力としては `read()` メソッドを持つ任意のオブジェクトを受け取り、具体的なインスタンス（実際のファイルオブジェクトなど）を返します。型ヒントでは Protocol を使って明確化することもあります。
- **TypeScript**: 柔軟性のために型パラメータやユニオン型を入力として受け取り、具象型を返します。たとえば `string | Buffer` を受け取って `string` を返す、といった形です。
- **Rust**: トレイトオブジェクトではなく `impl Trait` を入力として受け取り、具象型を返します。コンパイラがこの非対称性を強制します。

### トレードオフと制約

インターフェースを受け取ることで、呼び出し側に柔軟性を与えられます。モック化、差し替え、疎結合といったことが可能になります。

具象型を返すことで、呼び出し側はオブジェクトの持つ機能すべてにアクセスできるようになります。インターフェース経由のディスパッチに伴う実行時コスト（ほとんどの言語では無視できる程度ですが）を避けられますし、それ以上に大切なこととして、「このオブジェクトに対して実際に何が呼べるのか」という認知コストを下げられます。

トレードオフもあります。インターフェースを返す場合、呼び出し側はそのインターフェースが公開するメソッドのみを使うという前提で設計されます。具象型を返す場合、呼び出し側は文書化されていないメソッドに依存してしまうかもしれません。解決策はドキュメントです。「これらのメソッドは保証する。それ以外は実装の詳細である」と明示します。

ライブラリの文脈では、この非対称性がしばしば逆転します。実装の詳細を隠蔽して安定性を示すために、あえてインターフェースを返すのです。一方、アプリケーションコードでは、完全なオブジェクトが必要になる場面が多いため、具象型を返すのが自然です。

### 例

サービス内のロギングを例に取ります。

```go
// Go: インターフェースを受け取り、具象型を返す
func NewOrderService(log Logger) *OrderService { /* log はインターフェース */ }
func (s *OrderService) Log(msg string) error { /* nil か具象エラーを返す */ }

// これに対して、弱い設計の例:
func NewOrderService(log *SyslogLogger) *SyslogLogger { /* モックが使えなくなる */ }
```

テストでは、Logger インターフェースを実装したモック（適切なメソッドを持つ任意の型）を渡します。サービスは具象のエラー型、もしくは nil を返します。

- `testing-discipline` → 「Mocking vs. Integration」: インターフェースを受け取る設計が、なぜテスト可能なコードへの入り口になるのか。
- `review-taste` → 「Design Review Heuristics」: インターフェースが広すぎる／狭すぎると感じ取るための勘どころ。

## 拡充のルール

この節では、エージェントと開発者がドメインノートをどのように保守し、洗練していくのかを説明します。

### 新規作成・深化・精緻化・修正をどう判断するか

エージェントが学びを伝えるべき場面に出会ったときは、次の流れで判断してください。

1. **その話題に対応する概念の節はすでに存在するか？**
   - **存在し、かつ内容が正確**: 既存の節が取りこぼしているニュアンスや注意点がある場合は、「トレードオフと制約」あるいは「関連する節」に新しいサブ節として追記してください。
   - **存在するが内容が不完全**: その節を深化させてください（例を増やす、手順を明確化する、など）。改訂フラグは立てず、暗黙のうちに洗練していきます。
   - **存在するが内容が誤っている**: 「Prior Understanding (revised YYYY-MM-DD)」としてマークし、正しい内容に書き直してください。古い記述はマーク付きのサブ節に保存しておきます。
   - **存在しない**: 該当するドメインファイルに新しい概念の節を作成してください。

2. **その概念は既存のドメインに収まるか、それとも新しいドメインが必要か？**
   - 19 のドメインのいずれかの対象範囲に収まるなら、そのドメインを使ってください。
   - どのドメインにも収まらないとすれば、それは分類体系が不完全であるというサインです。新しいドメインを作成する前に、まずはチームで議論してください。そうしないと分類がどんどん細分化していってしまいます。

### 改訂履歴の残し方

節を修正する際には、古い理解をそのまま消すのではなく、概念の末尾に日付付きサブ節を作って残します。

```markdown
### Prior Understanding (revised 2026-04-20)

この節の以前のバージョンは、[古い主張] と述べていた。これは [理由] のため、誤っていた／不完全だった。修正後の理解は上記に反映されている。
```

こうすることで学びの軌跡が記録されます。数週間後に開発者が同じファイルを開き直したときに、何が変わり、なぜ変わったのかを追えるようになります。また、将来のセッションで同じ話題を再検討するときにも、この履歴が判断材料になります。

### クロスリンクの張り方

同じドメインファイル内では、Markdown の見出しアンカーを使って関連する概念へリンクします。

```markdown
[同じドメイン内の「集約の境界」も参照](#aggregate-boundaries)
```

別のドメインファイルを参照するときは、ファイル名を明示します。

```markdown
[非正規化がパフォーマンスに与える影響](../performance-intuition.md#caching-trade-offs)
```

こうしてノート全体を通じて概念のウェブが編まれていきます。別途インデックスを作る必要はありません。データモデリングを学ぶ開発者は、リンクをたどってパフォーマンスやエラーハンドリングの関連する節まで進んでいけます。

### 長く読まれるための文体と時制

ドメインノートは、読者が数ヶ月後に再び読み返したとき、当時のセッションの文脈なしで理解できるように書いてください。

- **文体**: 中立的で説明的に書きます。「私たちが議論した」「発見した」といったセッション固有の表現は避け、「このプロジェクトでは」「このパターンは」「調査から分かっていることは」といった表現を用います。
- **時制**: 普遍的な原理には現在形を用います（例: 「イベントソーシングは書き込みと読み込みを分離する」）。プロジェクト固有の判断には過去形を用います（例: 「このプロジェクトは 2025 年にパフォーマンス上の理由から PostgreSQL を選択した」）。
- **命名**: パターンや概念には明示的な名前を与えます。セッション固有の文脈がなければ意味を取れない代名詞は避けてください。「それのほうが良かった」ではなく、「イベントソーシングは可観測性を高める一方で、結果整合性というコストを持ち込む」のように書きます。

### 避けるべきアンチパターン

以下のような内容は含めないでください。

- **セッション固有の語り**: 「キャッシュミスをデバッグしていたときに気づいたのだが……」のような書き方。デバッグのエピソード自体は有益ですが、そこから一般原則を抽出して書き残すようにしてください。
- **内輪のネタや参照**: 「昨日の Slack のスレッドみたいに……」のような書き方。未来の読者はその Slack スレッドを持っていません。
- **原則を伴わないツールのバージョン番号**: 「Jest 27 から 28 にアップグレードしたら……」のような書き方。Jest そのものに関する学びであれば、それを明記してください。テスト分離のようにフレームワーク非依存の学びであれば、バージョン番号とは結びつけないでください。
- **すぐに古くなるリンク**: 「PR #1234 を参照」のような書き方。Issue はクローズされ、PR の番号はフォークのたびに変わり、リンクはすぐに腐ります。代わりに ADR や恒久的なドキュメントの節を参照してください。
- **個人情報や内部のシークレット**: 顧客名、社内のメールアドレス、プライベートな API キーなどを含めないこと。ドメインファイルはリポジトリに置かれます。
- **非難的・評価的な表現**: 「昔のコードはぐちゃぐちゃだった」のような書き方。個人や書き手を批判するのではなく、パターンそのものに焦点を当ててください。

## ノートに含めてはいけないもの

エージェントと開発者は、次のような内容を明示的に除外してください。

- **セッション固有のコミット SHA**: 「コミット abc123 を参照」のような書き方は、時間とともに腐ります。代わりに ADR や恒久的なドキュメントの節を参照してください。
- **一時的なツールのバージョン**: その学びがツール自体の挙動についてでない限り、バージョン番号に縛りつけないでください。
- **利用者の個人情報（PII）**: 顧客名、メールアドレス、社内の識別子などは含めないでください。
- **プライベートなシークレット**: API キー、パスワード、非公開の設定値などは含めないでください。
- **「現時点」を前提にした日付付きの記述**: 「2026 年 4 月時点でフレームワークは X をサポートしていない」と書いてしまうと、半年後には事実でなくなっているかもしれません。代わりに「フレームワークは歴史的に X を欠いていた。状況が変わった場合はこの節を更新すること」のように書いてください。
- **セッション固有のナレーション**: 「このセッションでは〜を決定した」といった表現。ドメインノートは時間を超えて通用する概念へと蒸留すべきです。セッション単位の記録は、各応答の「ノートブック diff」にのみ残ります。
- **非難や感情的な表現**: 「古いコードはひどかった」と書くのではなく、「以前のバージョンは [パターン] を採用していた。これは [理由] のため置き換えられた」と書いてください。
- **クイズや宿題形式の問いかけ**: 「イベントソーシングを使う理由を考えてみましょう？」のような書き方はしないでください。理由を明示的に説明します。
- **称賛や励まし**: 「このパターンに気づけて素晴らしい！」のような表現は避け、事実に徹してください。

## 他の Growth Mode ファイルとの関係

このドメイン分類体系のドキュメントは、**リファレンス** としての役割を担います。その周辺には、以下のファイルがあります。

- **`.claude/growth/config.json`**: 利用者の現在のレベルと、エージェントごとの適用範囲を保持する設定ファイル。知識を格納するファイルではありません。デフォルトで gitignore されます。
- **`.claude/growth/preamble.md`**: 15 の Growth 対応エージェントすべてが共有する「拡充の約束事」を記述したファイル。ここからこの分類体系のドキュメントを参照します。
- **`.claude/growth/notes/<domain>.md`**: 正典となる 19 のドメインファイルと、学習者が必要に応じて作成するカスタムドメインのファイル群。インストール時は最小限のプレースホルダだけがシードされ、セッションを通じてエージェントが拡充・深化・精緻化していきます。デフォルトで gitignore されており、共有したい場合は `.gitignore.example` を通じてオプトインで公開できます。

流れとしては次のとおりです。**エージェントが学びを伝えるべき場面に遭遇する → 対応するドメインファイルを読む → 追記・深化・精緻化・修正・新規ドメインの作成のいずれを行うかを判断する → 非破壊的に変更を適用し、応答のなかで差分を報告する → 将来のセッションで同じドメインファイルが開かれ、理解がさらに重ねられていく。**

時系列のジャーナルは存在しません。セッション単位の記録は、応答ごとの「ノートブック diff」が担います。そして長期的な監査証跡は、Git の履歴に委ねられます。

## バージョニングと進化

ドメインの分類体系そのものも、時間とともに進化します。

1. **ドメインの追加**: 複数のセッションを通じて新しい知識領域が立ち現れた場合（例: チームの連携に関する学びを蓄積する「team-scaling」など）。
2. **ドメインの統合**: 2 つのドメインが常に互いを参照しあっており、1 つにまとめたほうが明快になる場合。
3. **ドメインの分割**: ドメインが大きく育ちすぎて取り回しにくくなった場合（例: `architecture.md` が 10 以上の主要な節を抱えるようになり、読みこなしづらくなった場合など）。

分類体系の変更は、このファイルの改訂履歴に記録します。分類の変更がプロジェクトの優先順位の変化を示唆する場合には、ADR を作成します。

## はじめに: 最初のセッションで行うこと

Growth Mode を初めて有効にする開発者は、次の手順を踏んでください。

1. このドキュメントを一読し、19 のドメインが何を指しているのかを把握する。
2. `/growth on junior` を実行して Growth Mode を有効化する。
3. 普段どおりに作業する。エージェントが学びを伝えるべき場面に出会ったら、対応するドメインファイルに追記される。
4. セッションの終わりに、各応答の末尾に付く「ノートブック diff」を見て、どのドメインファイルが更新されたかを確認する。
5. 拡充されたドメインについて `.claude/growth/notes/<domain>.md` を開き、そこまでに蓄積された学びを読み返す。

19 のドメインファイルはインストール時に最小限のプレースホルダとして初期化され、セッションを重ねるにつれてエージェントの手で充実していきます。

## よくある質問

**Q: 2 つのドメインが重なって見えるときはどうすればよいですか？**

A: ドメインは成果物の種類ではなく関心事に基づいて整理しているため、ある程度の重なりは想定内です。たとえば `error-handling` と `testing-discipline` の双方が例外安全性に関心を持つ、といったケースです。その場合は相互にリンクを張ってください。2 つのドメインのあいだを頻繁に行き来する必要があり、1 つの概念としてまとめたほうが自然だと感じたら、Issue を立てて議論してください。統合すべきか判断しましょう。

**Q: ドメインファイルの保守は誰が担当しますか？**

A: 誰でも貢献できます。作業中のエージェント、セッションを振り返る開発者、パターンを見出すコードレビュアー、いずれもが担い手です。中央に管理者を置くのではなく、リポジトリ全体で保守していきます。PR では、コードの変更と同じようにドメインファイルの変更もレビューします。

**Q: 関係がなくなったドメインは削除できますか？**

A: ドメインはほとんど削除しません。代わりに、分類体系のなかで非推奨としてマークし、ファイルをアーカイブします。将来のプロジェクトやチームが、その知識に価値を見出すかもしれないからです。

**Q: エージェントのドメインの理解が誤っていた場合はどうしますか？**

A: ドメインファイルはエージェントの「唯一の正」ではなく、チームの現時点での理解を残したものです。エージェントや開発者が誤りに気づいたら、該当する節を修正し、古い内容は「Prior Understanding (revised YYYY-MM-DD)」としてマークして保存してください。ファイルは時間をかけて真実に近づいていきます。
