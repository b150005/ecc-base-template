## 1. 解くべき課題

ECC のエージェントチームを使う開発者は、完成度の高い成果物（コード、アーキテクチャドキュメント、テストスイート、セキュリティレポートなど）を手にします。ところが、それぞれの判断がなぜ下されたのか、どのようなトレードオフがあるのか、その選択がプロジェクト全体のパターンのなかでどう位置づけられるのか、といった情報は伝わってきません。implementer の出力をコピーして本番にデプロイすれば、開発のスピードは上がります。しかし、そこから理解の深さまで上がるとは限りません。

このギャップは成果物の品質の問題ではありません。implementer が返す優れた関数それ自体は、そこに組み込まれたパターン、エラーハンドリングの形を決めた境界条件、あるいはそのプロジェクトがなぜ慣用的な選択ではなく自明でない選択をとったのかを記録する ADR については、何も教えてくれません。半年後に別の文脈で同じパターンに出会った開発者は、初回に判断の根拠が示されなかったため、そのパターンを自力で再発見するか調べ直すしかありません。

こうした知識の取りこぼしは、大きく 3 つに分類できます。

- **パターンの認識不足**: 開発者はエージェントが生成したパターンを使いますが、そのパターンの名前、代替案、適用すべきでない条件までを自分のものにできません。再現はできても、応用のための推論はできません。
- **文脈の認識不足**: エージェントが既定から逸脱した選択をすることがあります。出力は正しく、レビューも通るため、開発者はその逸脱に気づきません。しかしその逸脱は重要な判断であり、次に同種の判断に直面したときに理解しておくべきポイントです。
- **知識の断片化**: 学びの機会はセッション全体に散らばっています。メモはここに、インラインコメントはあちらに、という状態で、新しいタスクで同じ概念に再会したときに参照できるリファレンスとしてはまとまりません。断片は、まとまった心的モデルへと統合されません。

Developer Growth Mode は、これら 3 つすべてに対処します。本機能は、宣言されたスキルレベルに合わせて判断の根拠を成果物と並べて提示し、その根拠を、開発者が実機能をデプロイしながら築き上げる、ドメインキー付きの知識ベースへと時間をかけて蓄積していく、オプトイン方式のアノテーション層です。成果物そのものを変更することはなく、作業をゲートすることもなく、明示的な操作なしに開発者のレベルを推論したり変更したりすることもありません。

既定状態（Growth Mode オフ）は、本機能を望まない開発者にとって、それまでとまったく変わりません。

## 2. ゴールと非ゴール

### ゴール

以下のゴールはいずれも検証可能です。各ゴールについて、観測可能な条件と測定方法を示します。

1. **全エージェントが対応する**: Growth Mode が有効なとき、チーム内の 15 エージェント全員が、宣言されたドメインの担当範囲に応じて知識ベースの拡充に参加する。測定方法: 15 エージェントすべてのファイルに含まれる `## Growth Domains` セクションを検査し、各エージェントに課された拡充のコントラクトが、そのエージェントの主要タスクを実行するセッションで実際に発動することを確認する。

2. **デフォルト OFF の不変条件**: Growth Mode が無効（＝デフォルト）のとき、どのエージェントの応答にも Growth Mode のアーティファクト（`## Growth:` 節、`[Growth Note]` マーカー）が現れず、`.claude/growth/notes/` 配下のファイルも書き込まれない。これはエージェントプロンプトの分岐ロジックに関する設計上の表明であり、出力のバイト一致に関する表明ではない（LLM の出力は実行ごと・モデルバージョン・プロンプト圧縮イベントを通じて非決定的であり、ゴールデンファイルに対するハッシュ突き合わせはアンチパターンである）。検証方法: `scripts/check-growth-invariants.sh` が、この不変条件を強制する 3 つの前提条件（`growth` Skill の `disable-model-invocation: true` フラグ、すべての Growth 対応エージェントプロンプトのガード分岐、gitignore の方針）を PR ごとにアサートする。

3. **知識ベースがドメイン単位で蓄積される**: 学びの機会が少なくとも 1 つ生じたセッションのあとに、対応するドメインファイル（`.claude/growth/notes/` 配下）には、新規または深化された節が現れる。観測方法: セッションの前後でノートディレクトリを diff し、少なくとも 1 つのドメインファイルが純粋に内容の追加を示すことを確認する。

4. **非破壊的な編集**: 拡充操作は、操作前に存在していたドメインファイルの内容を黙って削除することがない。これは `.claude/growth/preamble.md` に明記された設計目標であり、プリアンブルやエージェントプロンプトに変更が加わる PR のレビュー時に確認する。エージェント出力に対する自動アサーションでは検証しない。

5. **重大度の保全**: code-reviewer が指摘する問題は、Growth Mode のレベルにかかわらず本来の重大度で報告される。CRITICAL な指摘は、`junior`／`mid`／`senior` のいずれのレベルでも CRITICAL のラベルが付与される。これはエージェントプロンプト上の設計要件であり、code-reviewer のプロンプトに変更が加わる PR のレビュー時に確認する。フィクスチャベースの出力照合では検証しない。

6. **Skill がトグルの状態を正しく変更する**: `/growth on junior`、`/growth off`、`/growth status`、`/growth focus <domain>`、`/quiet` のいずれを呼び出しても、同一セッションのなかで、記録された設定変更または出力抑制の効果が得られる。測定方法: コマンド形式ごとに受け入れテストを用意する。

### 非ゴール

1. Growth Mode は、エージェントが生成したコードや成果物を一切、変更・ゲート・遅延・改変しない。本機能はアノテーションと拡充だけを行う層である。
2. Growth Mode は、振る舞いのシグナルから開発者のレベルを推測したり、自動的にエスカレーションしたりしない。宣言されたレベルこそがそのセッションにおけるレベルである。
3. Growth Mode は、本リリースではドメインノートの共有機能やエクスポート機能を提供しない。
4. Growth Mode は、進捗スコアリング、完了マーカー、バッジ、連続記録、その他のゲーミフィケーション要素を含まない。
5. Growth Mode は、セッションの内容、コミット履歴、その他の振る舞いに基づくプロキシ情報から、レベルを自動的に推論しない。
6. Growth Mode は、複数の開発者のあいだでノートをマージしたり同期したりしない。ドメインファイルはあくまで個人の学習成果物である。
7. Growth Mode は、本リリースにおいてはドメインノートの日本語訳や他言語への翻訳を提供しない。
8. Growth Mode は、オーケストレーターの委任ロジックには適用されない。オーケストレーターが Growth に対して負う責務は、フロントマターでドメインの担当を宣言すること、および自分自身の応答に学びの機会が含まれた場合に拡充プロトコルに従うこと、この 2 点に限られる。

## 3. ユーザーセグメントと Jobs-to-Be-Done

ここに示すのは、本機能がサービスを提供する対象となる利用者である。以下の表は、対象ユーザー層を記述したものであって、ビジネス上の成長ターゲットではない。

| セグメント | 説明 | JTBD（片付けたい仕事） |
|---------|---------|------|
| キャリアを切り替えた開発者 | 静的型付け、コンパイル型、あるいは強く慣用表現が根付いた言語に、はじめて本格的に取り組む。以前いた領域では十分な適性を発揮していた。 | エージェントが生成したコードを受け入れる際、それが体現しているパターンを理解したい。次の機能を実装するときに、単なるコピー＆ペーストではなく、意図して同じパターンを再現できるようになりたい。 |
| 新しいスタックに乗ったミドル層のエンジニア | 経験年数は 3〜5 年。プロジェクトのスタックやフレームワーク自体は未経験。プログラミングの基礎は押さえているが、このエコシステムにおける「慣用的」が何を指すのかはまだ掴めていない。 | エージェントが自分の書いたであろう選択から外れたとき、従っている規約の説明がほしい。作業を中断して調べ物をしなくても、素早く感覚を合わせられるようにしたい。 |
| チームメイトをオンボーディングするシニアエンジニア | コードベースの所有者。経験の浅い開発者とペアで作業しており、その相手もエージェントを使っている。 | エージェントが自明でないアーキテクチャ選択をしたとき、レビューの会話のなかで参照できるトレードオフの記録がほしい。そうすれば、議論を第一原理からではなく、共有された語彙から始められる。 |

この 3 つのセグメントが、本機能の想定利用者を定義する。本機能は、特定のチーム規模や企業フェーズ、あるいは Claude Code と ECC テンプレートの枠を越えたツールチェーン環境については想定していない。

## 4. レベルセマンティクス

Growth Mode には `junior`、`mid`、`senior` の 3 つのレベルがあります。レベルが制御するのは、ベースとなる文脈の切り口と密度（エージェントが開発者にどの程度の前提知識を仮定するか、およびどのような判断をノートに書き留める価値があるかの判断基準）です。レベルは冗長さを調整するツマミではなく、トークン予算やノート数の上限を設けるものでもありません。深さは概念そのものの性質であり、レベルはどの概念がノートとして書き留める基準を満たすかを決めます。

3 つのレベルのいずれも、同じドメインファイルに書き込みます。ノートブックがレベルごとに分岐することはありません。セッション 1 で書かれた junior レベルの基礎的な説明と、セッション 12 で書かれた senior レベルのトレードオフに関する洗練は、同じ節のなかに共存し、層状に並びます。上には基礎、中には慣用表現の差異、下にはトレードオフに関する考察、という具合です。

### junior

エージェントは第一原理から説明します。用語を使う前に、その用語を定義します。パターンには名前を付け、事前の予備知識なしに開発者が選びがちな素朴な代替案と対比させて示します。具体例は丁寧に展開されます。前提となる概念は、その場で説明するか、それを扱うドメインファイルへの相互参照として示します。エージェントは、開発者がその概念に何らかの文脈で触れた経験があるとは仮定しません。

junior レベルの貢献として典型的なのは、第一原理の説明、慣用表現のブロック、トレードオフのブロックを備えた一続きの節であり、複数段落になることもあります。その長さは意図されたものです。ここで基礎の足場を組み上げているからです。

### mid

エージェントは第一原理は既知とみなしますが、自明ではない点については説明します。別のスタックから移ってきた有能なエンジニアでは推測しづらい、このスタックに習熟した実践者が実際にとっている振る舞いに焦点を当てます。トレードオフには名前を付けますが、代替案を網羅的に比較することはしません。mid レベルの開発者であれば知っているはずの概念を繰り返し説明することはせず、このスタックやプロジェクトが一般的な経験から外れているポイントに絞って記述します。

mid レベルの貢献は、junior レベルのそれよりも短くなるのが典型的で、1〜3 段落程度に収まります。ただし、簡潔に済ませるわけではなく、やはり「なぜ」の部分は説明します。

### senior

エージェントは、判断が既定から外れている場合にのみ貢献します。ノートでは既定の選択肢を名指し、実際に採った選択肢を名指し、この文脈でそちらが優先された理由を述べます。senior レベルの貢献として書かれるのはトレードオフの記録であり、判断に固有で凝縮された内容です。まさに、書き手の頭のなかだけに残ってしまいがちな思考を掬い上げるがゆえに、価値があります。既定以外の判断が含まれないセッションでは、貢献されるノートは 0 件になります。これは失敗ではなく、正しい挙動です。

### レベルごとのサンプル

各サンプルは、実際のエージェント出力を模したものです。例として、architect、implementer、test-runner、code-reviewer、market-analyst、technical-writer、devops-engineer の 7 エージェントを取り上げ、19 ドメインの分類体系全体をカバーします。

#### architect — architecture ドメイン

**Junior:**
> ここではイベントソーシングの境界により、書き込みモデルは追記専用、読み取りモデルは結果整合として扱われます。この 2 つは独立したモデルであり、データストアを共有しません。正典となる説明は Martin Fowler の Event Sourcing パターンです。実プロジェクトにおいては、集約に求められる整合性要件が単純な CRUD 方式を排除した理由は `docs/en/adr/NNN-event-sourcing.md` のような ADR に記録されることになります（例示のためのパスであり、ベーステンプレートには同梱されません）。押さえておくべき要点は、「結果整合」は意図的に受け入れたトレードオフだということです。読み取りモデルはわずかに古いかもしれませんが、その代わりに書き込みモデルは、読み取りを捌くためにロックを取る必要がありません。

**Mid:**
> 腐敗防止層（Anti-Corruption Layer, ACL）は、アダプタ層ではなくドメイン境界に配置します。この差は重要で、アダプタ層に ACL を置くとアダプタ自身はきれいに保てますが、呼び出し規約を通じて外部スキーマがドメインオブジェクトへ漏れ出します。ドメイン境界に ACL を置けば、この漏れを完全に防げます。これはプロジェクトのスカフォールドの既定からは外れた判断であり、スカフォールドは ACL を省略していました。実プロジェクトでは、マッピング処理が 1 段増えるコストを正当化した外部 API の所有権に関する制約を、対応する ADR に記録しておくことになります。

**Senior:**
> ドメイン境界での ACL を、アダプタ層での変換より優先。コスト: すべての入力呼び出しでマッピング処理が 1 段増える。メリット: ACL を通さない限り、外部スキーマの変更がドメインオブジェクトに到達しない。既定は、外部 API が自チーム所有かつ安定している場合にこれを省略する。（実プロジェクトでは、ここで該当の ADR 番号を引用する。）

#### implementer — error-handling ドメイン

**Junior:**
> ここでは例外を投げる代わりに `Result<T, E>` を用いています。これは構文上の好みの問題ではなく、「エラーを誰が扱うのか」という問題です。関数が例外を投げる場合、呼び出し元はそれを捕捉する必要があることを自ら認識し、エラー経路を扱わなければなりません。この扱いは呼び出し側のコード上では見えません。一方、関数が `Result<T, E>` を返す場合、型システムが呼び出し元に「どちらの結果もあり得る」と気づかせ、エラー経路は呼び出しのまさにその場で明示的に処理されます。これは railway-oriented programming のパターンです。プロジェクトにおけるエラー境界の規約については、`docs/en/` とエラーハンドリング用のドメインノートを参照してください。

**Mid:**
> リポジトリのメソッドは ADR-007 に従い、`T | null` ではなく `Option<T>` を返します。違いはこうです。`Option<T>` は呼び出し元にアンラップを強制することで、値が存在しないケースを型システム上で明示的に扱わせ、null がモジュール境界を越えて広がるのを防ぎます。このコードベースには、まだ `null` を返している古いファイルが残っています。それらは移行対象です。モジュール境界のなかでは、どちらかに統一してください。同じモジュール内で null と Option を混在させるのは、どちらか一方に統一するよりもかえって悪い状態です。

**Senior:**
> ADR-007 に従い、`T | null` より `Option<T>` を優先。古いファイルはまだ移行中。モジュール境界内で 2 つの方式を混在させると、読み手に認知的な負荷を与える。変更を加えるタイミングで統一すること。

#### test-runner — testing-discipline ドメイン

**Junior:**
> テストは Arrange / Act / Assert（AAA）で構成し、Assert 部にはロジックを置きません。条件分岐もループも、アサーション自体以外のメソッド呼び出しも書きません。なぜ重要かというと、テストが失敗したとき、失敗箇所はアサーションの行を直接指し示すべきだからです。Assert ブロック内に分岐があって「どのアサーションを実行するか」を選ぶような構造になっていると、それが不可能になります。きれいな AAA 形式のテストは、それ自体が「何を証明しているのか」を語ってくれます。Assert 部でロジックを書きたくなったとしたら、それはそのテストが証明しようとしているものが多すぎる、あるいはセットアップが足りていないサインです。

**Mid:**
> フィクスチャはインラインで構築し、`beforeEach` を介して共有しません。複数のテストの要件を満たそうとして状態を抱え込んでいく共有フィクスチャは、結合の温床になります。テスト A がフィクスチャに求める要件を変えると、テスト B が黙って壊れる、という現象が起こります。個々のテストは独立に再現可能であるべきで、必要なものはすべてそのテストのなかで宣言されているべきです。インラインフィクスチャは記述がやや冗長になりますが、それは現実的な範囲のコストです。一方、育っていく共有フィクスチャの結合コストは、ずっと大きなものになります。

**Senior:**
> `beforeEach` よりインラインフィクスチャを優先。トレードオフ: 冗長さと引き換えに分離を得る。この 2 つのテストは必要なセットアップが異なるため、両方のケースを条件分岐で扱う共有フィクスチャは悪手になる。

#### code-reviewer — review-taste ドメイン

**Junior:**
> 14 行目の変更（ミューテーション）を CRITICAL として指摘します。影響はこの関数のなかにとどまりません。関数が引数をミューテートすると、呼び出し元が保持している参照の指す先が、呼び出し元の知らないうちに書き換わります。同期的で単一スレッドのフローのなかであれば追跡は可能ですが、非同期の境界をまたいだり、オブジェクトがコンポーネント間で共有されたりすると、そのミューテーションは見えなくなり、原因の特定が難しくなります。修正は、引数を書き換えるのではなく新しいコピーを返すことです。これは `rules/common/coding-style.md` に定められた不変性（immutability）ルールです。Growth 向けの補足: このルールが存在する理由は、隠れた状態変更が、大規模コードベースにおいて再現の難しいバグを生む最も一般的な原因の 1 つだからです。

**Mid:**
> 22〜31 行目のネストした条件分岐は、プロジェクトで定めた「ネストは 4 段まで」という上限を超えています。これは HIGH ではなく、スタイル上の指摘として扱います。深いネストがテストカバレッジの有効性を下げるのは、深くネストされた条件内の分岐にテストから独立して到達するのが難しいためです。推奨するリファクタリングは、内側の分岐を名前付きの述語として抽出することです。述語に名前を与えるのは装飾のためではなく、条件が持つ業務上の意味を記録するためです。

**Senior:**
> 14 行目のミューテーション、非同期コンテキスト、CRITICAL。不変性ルールに従って新しいコピーを返すように修正。22〜31 行目のネスト条件、HIGH、名前付き述語として抽出を推奨。ほかにマージをブロックする問題はなし。

#### market-analyst — market-reasoning ドメイン（新しく正典に加わったドメイン）

**Junior:**
> ここに記録された競争上のポジショニングに関する判断は、ターゲットユーザーの主要な Jobs-to-Be-Done において、「出力までの時間短縮」が「理解の深化」より優先されるという仮定に立っています。この仮定は、PRD でどの機能から打ち出すかの選び方に影響するため、明示的に名前を付けておく価値があります。JTBD のフレーム（Clayton Christensen によるフレームワーク）は、ユーザーが製品から離れる理由を予測するためのものです。ユーザーは、機能数の多い競合にではなく、同じ根本的な JTBD をより良く満たしてくれる競合へ流れます。JTBD を理解することは、「誤った成果物」を築いてしまう危険から身を守ることにつながります。

**Mid:**
> 市場セグメントの定義には、表明選好データ（アンケートなど）ではなく、顕示選好データ（ダウンロード数、Issue、テンプレートを利用しているパブリックリポジトリなど）を用いています。この区別は競合分析において重要です。表明選好は「X がほしい」という声に過度に引っ張られがちですが、顕示選好はユーザーが制約下で実際に選んだ行動を示します。このため、分析の根拠となるデータソースは引用して明記します。セグメントを更新する際も、アンケートなどで置き換えるのではなく、比較可能な顕示選好のシグナルに基づいて更新してください。

**Senior:**
> セグメントの境界は、より広い「AI ツールの利用者」ではなく、「Claude Code を本番プロジェクトで使っているチーム」に引く。狭い定義は TAM の推定値を下げるが、シグナル品質は高まる。拡張する場合には、既定とは異なる獲得モデルが必要になる。参考となる判断の記録: プロダクト計画ドキュメント内の ADR 相当の記述。プロジェクト採用に関する根拠データが変化したら、併せて更新すること。

#### technical-writer — documentation-craft ドメイン（新しく正典に加わったドメイン）

**Junior:**
> ここで採用している ADR のフォーマット（Context、Decision、Alternatives Considered、Consequences）は、恣意的に並べたものではありません。それぞれの節に固有の役割があります。Context は、読者が判断の下された状況を自分で再構成せずに済むようにします。Decision は、評価を交えずに選択そのものを述べます。Alternatives Considered は、その判断が唯一の選択肢ではなかったことを示し、後年の「なぜその選択を？」という議論の空転を防ぎます。Consequences は、良い帰結と悪い帰結の両方を記録し、将来の保守担当者がそのトレードオフがまだ成り立っているかを評価できるようにします。いずれかの節を省くと、ADR の長期的な有用性は下がります。このフォーマットは Michael Nygard の ADR 実践に由来し、オープンソースプロジェクトで広く採用されています。

**Mid:**
> この API のドキュメントは、リファレンス部（どのパラメータを取り、どのようなレスポンスが返るか）と概念部（なぜこのエンドポイントがこの形になっているのか）に分けています。分離は意図的なもので、リファレンスドキュメントは「何を呼び出すか」に答え、概念ドキュメントは「なぜこのエンドポイントが存在するのか」に答えます。両者を合体させると、良いリファレンスにも良い説明にもなりません。この分割は、将来の更新にも耐えます。パラメータ名の変更はリファレンス部にだけ影響し、エンドポイントの目的を変える再設計は概念部にだけ影響します。

**Senior:**
> 単一情報源パターンによって保守するバイリンガルドキュメント: `docs/en/` を正典とし、`docs/ja/` はそこへリンクを張りつつ翻訳を保つ。これにより、二重執筆によるドリフトを回避する。その代償は、変更のたびに翻訳のパスが 1 回分必要になること。代替案は (a) 両言語を並行執筆で維持する（ドリフトのリスクが高い）、(b) 英語のみにする（保守コストは下がるが到達性が低下する）。現行のモデルは、翻訳遅れがユーザーの不満につながらない限りは有効である。

#### devops-engineer — operational-awareness と release-and-deployment ドメイン

**Junior:**
> この段階の CI パイプラインでは、lint、テスト、ビルドを並列ジョブではなく、順序付きのゲートとして実行します。理由は、lint が失敗するとテストの出力にノイズが混じるためです。lint に引っかかるファイルはパース段階で失敗することがあり、テストが偽の失敗を出してしまう恐れがあります。最初に lint を通しておけば、テストが落ちたときにそれが lint 起因ではないと確信できます。この順序付けはデバッグ時の利便性のためのものです。コストは、すべて成功する PR の CI がほんのわずか遅くなることです。どちらの選択肢にも正当性はありますが、本プロジェクトはスループットよりも「失敗シグナルの明確さ」を優先しました。

**Mid:**
> Docker イメージのベースは alpine ではなく distroless を使います。トレードオフとしては、distroless にはシェルもパッケージマネージャもデバッグツールも含まれないため、攻撃対象とイメージサイズを抑えられる一方で、ライブデバッグはやりにくくなります。構造化ログを集約基盤に出力しているプロダクションサービスであれば、コンテナに対してシェルで exec する必要は通常ありません。デバッグの入り口はログです。もし運用モデルがコンテナに `exec` することを前提としているのであれば、alpine（またはデバッグ用のサイドカー）のほうが適切な選択となります。このプロジェクトの選択は記録してあり、運用モデルが変わった時点で見直すべきです。

**Senior:**
> SIGTERM でグレースフルシャットダウン。SIGKILL までのドレイン期間は 15 秒。この値は直近の負荷テストにおける 95 パーセンタイルのリクエストレイテンシの 2 倍を目安に設定している。レイテンシ分布が変わったら、ドレイン期間も更新すること。この設定は Dockerfile ではなく Kubernetes のデプロイメントマニフェストに記述されているため、双方を同期させ続ける必要がある。

## 5. 機能要件

### FR-001: Skill の定義

Growth Mode は Claude Code の Skill として切り替えられる。Skill ファイルの置き場所は次のとおり。

```
.claude/skills/growth/SKILL.md
```

Skill の YAML フロントマターには、以下を必ず含めなければならない。

```yaml
---
disable-model-invocation: true
arguments: [action, level]
---
```

`disable-model-invocation: true` は、Skill がユーザー操作（`/growth ...`）からのみ起動でき、モデル自身が自動で呼び出せないことを保証する。

`arguments` フィールドは位置引数のトークンをマッピングし、`/growth on junior` を `$action=on`、`$level=junior` へと解決する。

Skill 本体は、コマンドのディスパッチ処理を実装する。すなわち、`.claude/growth/config.json` を読み、要求された状態変更を適用し、更新後の設定を書き戻す。初回の呼び出しでは、ファイルが存在しない場合に新規作成する。

Skill は、Growth Mode の状態変更が通る唯一の入口である。エージェントは設定ファイルを読み取りはするが、直接書き込むことはない（例外はノート拡充の処理だが、これは設定ファイルではなくドメインファイルに書き込む）。

### FR-002: Skill が受け付けるコマンド形式

以下の呼び出し形式をすべてサポートしなければならない。

| コマンド | 挙動 |
|---------|--------|
| `/growth on` | 設定に保存されたレベルで有効化する。保存されていない場合は `junior` を既定とする。|
| `/growth on junior` | `junior` レベルで有効化する。設定に `enabled: true`、`level: "junior"` を書き込む。|
| `/growth on mid` | `mid` レベルで有効化する。|
| `/growth on senior` | `senior` レベルで有効化する。|
| `/growth off` | 無効化する。`enabled: false` を設定する。次回の有効化に備えて `level` と `focus_domains` は保持する。|
| `/growth status` | 現在の有効／無効状態、レベル、`focus_domains`、直近 10 件のノート diff レポートの要約を表示する。設定は変更しない。|
| `/growth focus <domain>` | `focus_domains` を 1 つのドメインに設定する。エージェントは学びの機会をこのドメインで優先的に扱う。|
| `/growth focus <domain>,<domain>` | `focus_domains` をカンマ区切りの複数ドメインで設定する。|
| `/growth focus clear` | `focus_domains` をクリアする。エージェントはすべてのドメインを等しく扱う。|
| `/growth level junior\|mid\|senior` | 有効／無効状態を変えずにレベルだけを変更する。|
| `/growth domain new <key>` | 学習者に新しいカスタムドメインキーの確認を促し、確認が取れた時点でシード済みのドメインファイルを作成し、設定を更新する。|
| `/quiet` | 直後のエージェント呼び出しに対する 1 回限りのトレーラー抑制を行う。設定は変更しない。|

不明なサブコマンドについては、短い使用方法メッセージを返すのみでセッションは中断しない。

### FR-003: 設定ファイルのスキーマ

Growth Mode の状態は `.claude/growth/config.json` に永続化する。スキーマは次のとおり。

```json
{
  "enabled": false,
  "level": "junior",
  "focus_domains": [],
  "updatedAt": "2026-04-22T00:00:00Z"
}
```

各フィールドの意味。

- `enabled`: 真偽値。必須。ファイル自体が存在しない場合やパースに失敗した場合は `false` として扱う。
- `level`: 文字列。`"junior"`、`"mid"`、`"senior"` のいずれか。`enabled` が `true` のときには必須。
- `focus_domains`: ドメインキーの文字列配列。空でも構わない。空でない場合、エージェントは、リストに含まれないドメインで学びの機会に出会った際、それが理解の根幹に関わるのであれば貢献し、そうでなければ浅いエントリを残すのではなく貢献を見送る。
- `updatedAt`: ISO 8601 形式のタイムスタンプ。Skill が書き込みを行うたびに更新する。

ファイルは有効な JSON である必要がある。パースエラーは `enabled: false` として扱う。未知のキーは、書き込み時にもそのまま保持する（将来の互換性のため）。

このファイルは最初の `/growth on` 呼び出し時に作成される。最初の使用までは、リポジトリ内には存在しない。

### FR-004: 状態の永続化とエージェントの読み込みパス

設定は、セッション開始時にすべての Growth 対応エージェントが読み込む。読み込みの順序は次のとおり。

1. `.claude/growth/config.json` を読む。ファイルが存在しないか `enabled: false` の場合は、Growth に関するすべてのステップをスキップする。プリアンブルの読み込みも、ドメインファイルの読み込みも行わず、`.claude/growth/notes/` 配下のどのファイルも変更しない。
2. `enabled: true` の場合は、拡充プロトコルを確認するために `.claude/growth/preamble.md` を読む。
3. エージェント自身の `## Growth Domains` セクションに記述された からドメインキーを引き、現在のタスクに関連するドメインファイルを特定する。
4. それらのドメインファイルを読み込み、すでに記録されている内容を把握して重複を避ける。
5. 主要タスクを進める。学びの機会が生じたら、拡充プロトコルに従う。

この読み込みパスは各エージェントが独立に実行する。オーケストレーターがパース済みの設定を事前に配布することはしない。各エージェントは、自身で設定を読み込む責務を持つ。

### FR-006: エージェントごとのドメイン担当

### FR-007: 拡充操作のコントラクト

学びの機会が生じた際に、Growth 対応エージェントが従うべきコントラクトである。正典の定義は `.claude/growth/preamble.md` に置く。各エージェントはそのファイルをパスで参照するだけで、中身をインラインに展開しない。

5 ステップのコントラクトは次のとおり。

1. **対象ドメインを特定する**: 学びの機会を、`## Growth Domains` セクションにあるドメインキーへ対応づける。学びの機会が 2 つのドメインにまたがる場合は、概念がより基礎的な位置を占めるドメインを選び、副次のドメイン側には相互参照の 1 行リンクを置く。既存のどのドメインにも収まらない場合には、新しいドメインを提案し、学習者の確認（`/growth domain new <key>`）を待つ。ドメインファイルを自動で作成することはしない。

2. **対象のドメインファイルを読み込む**: エージェントは、どのように貢献するかを決める前に、必ず既存のファイル内容を読む。これは妥協の余地のない手順である。現状を読まずに書き加えると、内容の重複、以前のエントリとの静かな矛盾、既に居場所のある概念の断片化などのリスクを抱え込むことになる。

3. **操作を選ぶ**: 次のいずれか 1 つを選ぶ。
   - **Add**: まだ存在しない概念のために、新しい最上位の節を作成する。
   - **Deepen**: 既存の節に対して、新しい例、但し書き、エッジケース、相互参照などを追加する。既存内容は変更しない。
   - **Refine**: 既存節の表現を引き締めたり、例をより良くしたりする。主張そのものは変えない。元の表現は `Prior Understanding (revised YYYY-MM-DD)` のブロックで保存する。
   - **Correct**: 以前のエントリを supersede されたものとしてマークし、正しい理解をその下に書き加える。supersede された記述は削除しない。`> Superseded YYYY-MM-DD: <reason>` のマーカーとともに残す。
   - **New domain**: ステップ 1 で提案し、学習者の確認を得たあとにのみ実行する。

4. **変更を非破壊的に適用する**: 変更対象の外にある既存の見出し、例、コードブロックは、バイト単位でそのまま保つ。エントリが削除されることはない。ファイルが整理上の閾値を超えて大きくなった場合（分割の閾値については「オープンクエスチョン」の節を参照）、エージェントは差分レポートのなかで閾値超過をフラグとして報告する。エージェント単独で再編成を行うことはしない。

5. **差分を報告する**: 応答の末尾に、エージェントはチャット上でのみ可視な 2 つの節を出力する（いずれのファイルにも書き込まない）。

```
## Growth: taught this session
- [concept-name]: [宣言されたレベルに即した 1 文要約]

## Growth: notebook diff
- notes/<domain>.md → <operation> on `## <section-heading>`: <変更内容の 1 文要約>
```

これらの節は、ノートブックの変遷を監査するために開発者が参照する由来情報として機能する。また、これらの見出しの有無は、デフォルト OFF 時の手動スポットチェックにおいて確認するシグナルでもある。Growth Mode が OFF のときには、これらのヘッダは必ず出現しない状態でなければならない。

### FR-008: CLAUDE.md のポインタ

`.claude/CLAUDE.md` には `## Developer Growth Mode` の節を必ず含めなければならない。この節は無条件で配置され、Growth Mode が現在有効か無効かに関係なく、CLAUDE.md のなかに常に存在する。その内容は次のとおり。

```markdown

築き上げた個人用のリファレンスへと成長していきます。
設定ファイルは `.claude/growth/config.json` にあります。すべてのエージェントが従う拡充プロトコルは
`.claude/growth/preamble.md` に定義されています。現在の状態を確認するには `/growth status` を実行してください。
```

CLAUDE.md には、Growth Mode 向けのエージェントプロンプト指示は置かない。Growth Mode 関連の内容は、このブロック以外には CLAUDE.md のどこにも置かない。

エージェントは、セッション開始時にこのブロックを介してノートディレクトリと設定ファイルのパスを発見する。ブロックは十分に短いため、デフォルト OFF のセッションに対する文脈上のオーバーヘッドを実質的に増やすことはない。

### FR-009: ノートの Git 上の扱い

`.claude/growth/notes/` はデフォルトで gitignore 対象とする。リポジトリは次の状態で出荷される。

1. `.gitignore` のエントリによって、`.claude/growth/notes/` と `.claude/growth/config.json` がバージョン管理から除外される。
2. `.gitignore.example` ファイル（または既存の `.gitignore` 内のコメントブロック）が、開発者がノートをコミットすることを選んだときに編集すべき行を正確に示す。

デフォルトで gitignore とする根拠: ドメインノートには、開発者の誤り、以前の理解、supersede された履歴などが含まれる。これは私的な学習データである。ノートを共有したい開発者（たとえば、チームでの学習の取り決めや公開の研究ジャーナルとして）は、`.gitignore` を明示的に編集することで選択する。リポジトリ側が既定で共有を強制することはしない。

README では、デフォルトで非公開という経路と、オプトインで共有するという経路の両方を記述する。どちらが望ましいかについて README では論じない。

`config.json` は常に gitignore 対象とする。個人ごとのレベルとフォーカス設定を含んでおり、個人設定であってコミットすべきものではないためである。

### FR-010: 1 回限りのトレーラー抑制

`/quiet` は、直後のエージェント呼び出しにおける Growth のアノテーションを抑制する。`config.json` は変更しない。抑制された呼び出しのあと、次にエージェントが呼び出されたときには、通常の Growth の挙動が再開する。`/quiet` フラグは、任意のエージェント呼び出しの形式に付加できる。

### FR-011: フォーカスドメインによる優先づけ

`config.json` の `focus_domains` が空でないとき、エージェントはこれをソフトな優先シグナルとして扱う。フォーカスドメイン内で学びの機会に出会ったエージェントは、通常どおり完全な拡充エントリを書く。フォーカスドメイン外で学びの機会に出会ったエージェントは、その機会が理解の根幹に関わるかどうかを判断する。根幹に関わるのであれば通常どおり書き込む。そうでない場合には、浅いエントリを残すのではなく貢献を見送る。これは優先づけであり、強制的なフィルタではない。フォーカスリストに含まれていないからといって、本当に重要な学びの機会を抑え込むことはしない。

## 6. 非機能要件

### NFR-001: デフォルト OFF の不変条件

`.claude/growth/config.json` が存在しない、`enabled` が `false`、あるいはファイルをパースできない場合、いずれのエージェントの応答にも、以下の Growth Mode のアーティファクトが現れてはならない。

- エージェントの応答に `## Growth:` の節が現れない。
- `[Growth Note]` のマーカーも現れない。
- `.claude/growth/notes/` 配下のファイルは作成も変更もされない。
- `.claude/growth/preamble.md` の読み込みも発生しない。

これは本機能の根幹をなす要件である。エージェント応答に対する出力一致のアサーションではなく、3 つの決定論的な前提条件によって強制する（LLM の出力は非決定的であり、ゴールデンファイル回帰テストは遅かれ早かれ不安定テストに劣化する）。

1. `.claude/skills/growth/SKILL.md` の `disable-model-invocation: true`。
2. すべての Growth 対応エージェントプロンプトに含まれるガード分岐。`config.json` を読み、存在しないか無効であれば Growth 関連のステップをすべてスキップする。
3. `.claude/growth/notes/` と `.claude/growth/config.json` は gitignore の対象であり、誤って有効化された際の成果物がコミットに混入しない。

3 つとも、PR ごとに `scripts/check-growth-invariants.sh` で検証する。スクリプトは grep ベースの静的チェックのみを用い、LLM を介在させない。

### NFR-002: 成果物をブロックしない UX

Growth のアノテーションとノートの拡充は、本来の成果物をブロックしたり遅延させたり先行したりすることは決してない。成果物（コード、設計ドキュメント、レビュー、セキュリティレポート）は、常に応答の先頭にある実質的な内容である。Growth に関する節は常にその後ろに続く。開発者が次のエージェント呼び出しに進む前に、Growth の内容を確認したり応答したりする必要はない。応答の前にクイズを出したり、理解度を確認したりすることはしない。

### NFR-003: 深さに見合った説明量

ノートの長さ、ノートの件数、Growth の内容が消費するトークン数には、恣意的な上限を設けない。原則は、「概念が必要とする分だけ体系的に説明し、かつ、目の前のコードに可能な限り即した具体性を持たせる」というものである。字数を削るために「X を使わないとき」のケースを省くくらいなら、上限にとらわれず長く説明するほうを選ぶ。

許容されないのは、成果物の内容を散文で繰り返し要約すること、すでにドメインファイルに書かれている内容を重複して書くこと、開発者が読み飛ばしてしまうような埋め草的な説明である。長さは、中身によってのみ正当化される。

### NFR-004: 重大度の保全

code-reviewer が Growth Mode で出力する内容は、指摘の重大度ラベルを和らげたり、ヘッジしたり、条件付きにしたり、薄めたり、希釈したりしてはならない。CRITICAL はどのレベルでも CRITICAL のままである。Growth Note は、CRITICAL な指摘がなぜ CRITICAL なのかを説明することを許され、`junior` レベルではむしろ推奨されるが、その説明が重大度ラベルを変えることはない。Growth の節は指摘に対する「加算」であり、置き換えでも柔らかな言い換えでもない。

### NFR-005: プレッシャー下での非破壊的編集

非推奨となったエントリや supersede されたエントリに遭遇したエージェントは、必ず `correct` 操作を適用する。非推奨としてマークし、訂正版をその下に書き加える。沈黙のまま上書きすることはしない。エージェントはドメインファイルから以前の内容を削除してはならない。以前のエントリを要約したり圧縮したりして紙面を節約することもしてはならない。supersede されたエントリは、マーカーとともに積み上がっていく。これは設計であって、最適化すべき制約ではない。

### NFR-006: 人間が読むための Markdown、概念による整理

ドメインファイルは標準の Markdown で記述する。独自フォーマットは導入しない。整理の単位はあくまで概念（H2 見出しに概念の名前を記す）であり、日付やセッションではない。6 ヶ月後に開発者が `testing-discipline.md` を開いたとき、そこには時系列のログではなく、題材ごとに整理された一貫したリファレンスが現れているべきである。

### NFR-007: 同じドメインへの書き込みの直列化

オーケストレーターが同一のワークフローのなかで 2 つ以上のエージェントに委任し、それらエージェントの Growth Domains が重なっている場合、Growth への書き込みフェーズは順次実行する。Growth Domains が重ならない場合には、並列実行して構わない。これは、ノートブックに対する「読み込み→変更→書き戻し」の不変条件を保つためである。直列化のルールは、オーケストレーターのエージェントプロンプトに組み込む。

## 7. 参照する知識の層

Growth Note とドメインファイルのエントリは、3 つの層の知識を典拠にする。エージェントは、その場面に最も適した層の情報源を選ぶ。引用は、開発者が検証可能な形で示されていなければならない。

### 層 1: 外部の正典ドキュメント

公式ドキュメント、言語仕様、広く知られた名前付きのパターン（Martin Fowler のパターンカタログ、Gang of Four、OWASP、RFC 仕様、フレームワーク公式ドキュメントなど）へのリンクが該当する。層 1 の引用を行う貢献には、URL または名前付きパターンの記述を必ず含める必要があり、開発者がそれを検証できる形でなければならない。層 1 は、このコードベース以前から確立されている規約の話題を扱うときに優先される。

新鮮さに関する制約: エージェントは、固有の外部 URL を示せるときには学習データから正典のドキュメントを合成すべきではない。URL が古くなっている可能性がある場合には、検索した日付を明示するか、検証が必要である旨を注記する。古い情報や捏造された引用は、引用がまったくない状態より悪い。

### 層 2: プロジェクトの ADR

`docs/en/adr/` に記録された内容が該当する。層 2 の引用を行う貢献は、ADR を番号と短いタイトルで参照する（例: 「ADR-007: use-option-over-null」）。存在しない ADR への層 2 の引用を、エージェントが捏造してはならない。判断が正式に記録されていない場合には、該当の推論がローカルな規約である旨を注記したうえで、ADR 化の候補としてフラグを立ててもよい。

### 層 3: これまでのドメインノート

以前のセッションによって `.claude/growth/notes/` にすでに書き留められているエントリが該当する。エージェントは、以前のセッションで導入済みの概念を参照するときにこれを用い、現在のセッションではそこに詳細を足すか、相互参照を付け加える形で貢献する。層 3 は 3 つの層のなかで最も弱い権威しか持たないが、すでに存在する記述のうえに積み重ねていくために用いる。相互参照には相対 Markdown リンクを使う。

### 生成してはならないもの

- 何らかの `docs/growth/` ディレクトリツリー。
- 生成された Wiki ページ、学習モジュール、カリキュラムファイル。
- Growth の操作のなかで `.claude/growth/` の外に作られるファイル（ドメインファイルは `.claude/growth/notes/` 配下、プリアンブルと設定ファイルは `.claude/growth/` 配下に置く）。
- 教育目的のためだけに本番コードのファイルへ追加されるインラインコメント。
- ドメインファイル中のセッション固有の文脈記述（「キャッシュミスをデバッグしていたときに……」など）。ドメインファイルには抽出された原理を書くのであって、セッションログではない。

## 8. デフォルト OFF の不変条件

### 保証される内容

`.claude/growth/config.json` が存在しないとき、ファイルが `"enabled": false` を含むとき、あるいはファイルが有効な JSON でないときには、いかなるプロンプトに対するエージェントの応答にも、NFR-001 で定義した Growth Mode のアーティファクト（`## Growth:` の節、`[Growth Note]` マーカー、`.claude/growth/notes/` への書き込み）は含まれてはならない。

これはエージェントプロンプトのロジックに関する表明であり、出力のバイト一致に関する表明ではない。LLM の出力は、実行ごと・モデルバージョン・プロンプト圧縮イベントを通じて非決定的である。エージェント応答をゴールデンファイルに対してハッシュで照合しようとすれば、不安定テストに劣化して無効化されてしまう。本設計ではこの問題に対処するため、不変条件の強制を出力比較ではなく、決定論的で LLM を介在させないチェックに置く。

### 強制

以下の 3 つの決定論的な前提条件で不変条件を強制する。3 つとも PR ごとに `scripts/check-growth-invariants.sh` で検査する。

1. **Skill フラグの検査**: `.claude/skills/growth/SKILL.md` が `disable-model-invocation: true` を含んでいること。これにより、モデルが勝手に `enabled: true` に切り替えることを防ぐ。grep 1 行で検証可能。
2. **エージェントのガード分岐の検査**: `.claude/agents/` 配下で `## Growth Domains` セクションを宣言しているすべてのファイルに、ガード分岐のテキスト（`config.json` を読み、ファイルが存在しないか無効であれば Growth ステップをすべてスキップする）が含まれていること。エージェントファイルごとに正規表現 1 つで検証する。
3. **gitignore の方針の検査**: `.gitignore` が `.claude/growth/notes/` と `.claude/growth/config.json` を無視していること、および `.gitignore.example` がオプトイン反転のコメントブロックを含んでいること。grep 2 回で検証する。

3 つのいずれかが失敗すると CI ジョブが失敗し、その PR はマージできない。

### PR レビューで担保する設計目標

以下の 3 つの性質は、自動チェックではなく設計目標として扱う。エージェント出力に対して自動テストを行おうとすると、同じ非決定性の問題に突き当たるからである。`.claude/growth/preamble.md` やエージェントプロンプトに変更が加わる PR のレビュー時に、人間のレビュアーが確認することで担保する。

- **非破壊的な編集**: `deepen` 操作は対象節をバイト単位で保持し、途中への差し込みではなく末尾への追記を行う。`refine` 操作は元の版を supersede ブロックとして残す。`correct` 操作は、supersede マーカーの下に supersede されたテキストを可視状態で残す。
- **supersede の履歴**: 同じ節に対して `correct` 操作を複数回適用しても、マーカー付きですべての版が蓄積され、どの版も黙って削除されない。
- **重大度の保全**: code-reviewer の CRITICAL な指摘は、どのレベルでも CRITICAL のままである。

これらは `.claude/growth/preamble.md` および code-reviewer のエージェントプロンプトに明記されている。これらのファイルに変更が加わる際には、PR のレビュアーがこの性質が保たれていることを確認する。

## 9. 受け入れ基準

この機能が出荷可能と判断されるためには、以下のすべての基準を満たす必要がある。基準は QA で実行可能な形で記述する。

### Skill と設定

- [ ] `/growth on junior` を実行すると、`.claude/growth/config.json` が作成され、`enabled: true`、`level: "junior"`、`focus_domains: []`、および有効な `updatedAt` タイムスタンプが書き込まれる。
- [ ] `/growth off` を実行すると、設定の `enabled` が `false` になり、同一セッション内のそれ以降のエージェント応答には `## Growth:` 節が含まれない。
- [ ] `/growth status` は、現在の有効／無効状態、レベル、`focus_domains`、直近 10 件の差分レポートの要約を報告するのみで、ファイルを変更しない。
- [ ] `config.json` が存在しない、あるいは壊れている場合には、設定は無効として扱われ、開発者に対してエラーは表面化しない。エージェントのその後の挙動は、Growth Mode が存在しない場合と変わらない。
- [ ] `/growth on` をレベル引数なしで実行した場合、直前に保存されていたレベルを用いる。保存されたレベルがなければ `junior` を既定値として用いる。
- [ ] `/growth level senior` を実行すると、設定の `level` フィールドのみが変更され、`enabled` フィールドは変化しない。
- [ ] `/growth focus architecture,testing-discipline` を実行すると、設定の `focus_domains` が `["architecture", "testing-discipline"]` になる。
- [ ] `/growth focus clear` を実行すると、`focus_domains` が `[]` にリセットされる。
- [ ] `/growth domain new observability` を実行すると、確認プロンプトが表示され、確認後に `.claude/growth/notes/observability.md` がシード内容とともに作成される。

### 全エージェントの対応

- [ ] `.claude/agents/` 配下のすべてのエージェントファイルに、プロンプト本文冒頭の `## Growth Domains` セクションが存在する。
- [ ] architect が呼び出され、自明でない設計判断が関係するセッションでは、`architecture.md` または `api-design.md` に対して少なくとも 1 件の拡充が発生する。
- [ ] security-reviewer が呼び出され、セキュリティに関する指摘があるセッションでは、`security-mindset.md` に対して少なくとも 1 件の拡充が発生する。
- [ ] devops-engineer が呼び出され、デプロイに関わるパターンが用いられるセッションでは、`operational-awareness.md` または `release-and-deployment.md` に対して少なくとも 1 件の拡充が発生する。
- [ ] market-analyst が `junior` レベルで呼び出されたセッションでは、`market-reasoning.md` に対して拡充が発生する。
- [ ] technical-writer が呼び出されたセッションでは、`documentation-craft.md` に対して拡充が発生する。

### 拡充操作

- [ ] `add` 操作では、新しい概念のために新しい H2 の節が作成され、既存の節は変更されない。
- [ ] `deepen` 操作では、既存の節に追記が行われ、既存のテキストはバイト単位で変化しない。
- [ ] `correct` 操作では、以前のエントリに `> Superseded YYYY-MM-DD: <reason>` のマーカーが付き、訂正版がその下に追加される。supersede されたテキストもファイル内に残る。
- [ ] `refine` 操作では、元の表現が `Prior Understanding (revised YYYY-MM-DD)` ブロックに保存され、新しい表現が本来の位置に現れる。
- [ ] いかなる拡充操作も、ドメインファイルの内容を削除しない。
- [ ] エージェントは新しいドメインを差分レポートで提案する。`/growth domain new <key>` による確認を経ない限り、ドメインファイルが自動で作成されることはない。

### ノート差分レポート

- [ ] 拡充が行われた後、エージェントの応答は `## Growth: taught this session` と `## Growth: notebook diff` の 2 節で締めくくられる。
- [ ] 差分レポートには、ドメインファイル、操作、節の見出し、1 文の変更要約が含まれる。
- [ ] 差分レポートは、Growth Mode が OFF のときには出力されない。
- [ ] 差分レポートは、当該呼び出しに対して `/quiet` が指示されたときには出力されない。

### レベル

- [ ] `junior` レベルにおいて、architect が自明でない設計タスクに応答するとき、応答の教育的な節にはパターン適用の第一原理からの説明が完全な形で含まれる。
- [ ] `senior` レベルにおいて、implementer が既定以外の選択を含まないタスクに応答するとき、貢献されるノートは 0 件となる（応答本文と差分レポートのいずれにおいても）。
- [ ] `mid` レベルにおいて、code-reviewer は自明でない規約に関する指摘を説明する。日常的な整形に関する指摘には Growth の内容は付かない。
- [ ] Growth の内容はレベルによって切り口と密度が変わるが、主要成果物の正確さと完全性は変化しない。

### アンチパターン

- [ ] Growth の内容には、称賛の表現（「良い質問です」「よくできました」「素晴らしい」など）が含まれない。
- [ ] Growth の内容は、開発者に向けたクイズや問いかけの形を取らない。
- [ ] Growth の内容は、推論を加えずに成果物の内容を単に言い換えただけの散文にならない。
- [ ] CRITICAL とされた code-reviewer の指摘は、3 つのレベルすべてで CRITICAL のラベルを保つ。Growth の節がそのラベルを和らげたり条件付きにしたりすることはない。
- [ ] Growth 関連のファイルは `.claude/growth/` の外に作成されない。
- [ ] 本番コードのファイルに、教育目的だけのインラインコメントが追加されることはない。

### 呼び出し単位のトレーラー抑制

- [ ] `/quiet` は、直後の呼び出しにおける `## Growth:` の節およびノートの差分表示を抑制する。次の呼び出し以降は通常の挙動に戻る。

## 12. オープンクエスチョン

以下の問いは、本 PRD のなかでは結論を出さない。それぞれ、後続の ADR で扱うか、今後のイテレーションで取り扱う対象となる候補である。`[PRD-scope]` が付いている項目は、本ドキュメントの機能要件または非機能要件に影響を与えるもので、ADR で結論が出た際には PRD の修正が必要となる。`[ADR-scope]` が付いている項目はアーキテクチャ上の問題であり、後続の ADR で扱う。

1. **ドメインファイルはどのサイズで分割すべきか？** `[ADR-scope]` ADR における暫定的な立場は、1200 行もしくは最上位の節が 8 個を超えたときのいずれか早いほう、かつ technical-writer が `/growth review` でオンデマンドに発火させる形、というものである。本 PRD はその立場にしたがい、ADR で結論が出るまで要件としては正式化しない。

2. **market-analyst と monetization-strategist は `market-reasoning.md` と `business-modeling.md` に書き込むべきか？** `[CLOSED]` 全エージェント参加を優先する形で決着した。分類体系（担当対応表の主担当 ✓）と ADR-001（判断 1 により、リリース時点で 15 エージェント全員が Growth に対応。エージェント別表において market-analyst に `market-reasoning`、monetization-strategist に `business-modeling` が主担当として割り当てられている）の双方が、これらのエージェントが該当ドメインに書き込むことを定めている。本 PRD の FR-006 にも反映済みであり、さらなる議論は不要。

3. **ノートディレクトリはコミットすべきか、それともデフォルトで gitignore すべきか？** `[CLOSED]` デフォルト gitignore という形で決着した。ADR-001 の判断 4 と、同 ADR の「プライバシーの方針」の節が、いずれもこの方針を明確に述べている。本 PRD の FR-009 も同じ立場を反映している。ノートを共有したいチームには、`.gitignore.example` のオプトイン経路が仕組みとして用意されている。さらなる議論は不要。

4. **プロジェクトの途中でレベルを切り替えた場合、既存のドメインノートに何が起きるか？** `[ADR-scope]` ADR における暫定的な立場は、「新しい貢献は新しいレベルに従い、既存のエントリは書き換えない」というものである。`junior` セッションで書かれた基礎的な足場はそのまま残り、その上に `senior` での洗練が層として積み重なっていく。本 PRD もこの立場を引き継ぐ。

5. **並列エージェント実行との相互作用**: `[ADR-scope]` ADR において、Growth ドメインが重なる場合の直列化ルールが規定されている（本 PRD では NFR-007）。オーケストレーターが、ディスパッチの前に正確にドメインの重なりを検出する方法は、実装上の課題として残る。重なりを検出する具体的な仕組みは、まだ詳細まで詰めていない。

6. **Growth ノートの多言語対応**: `[PRD-scope]` ノートは本リリースでは英語のみとする。チームの主要な作業言語が英語でない場合、この制約は有効性を制限する。機械翻訳については本 PRD の範囲外であり、明示的にスコープ外とする。

7. **既存の CI フックとの相互作用**: `[ADR-scope]` プロジェクトの `PostToolUse` フックは、編集後に lint と型チェックを実行する。Growth Mode は `.claude/growth/notes/` 配下のドメインファイルに書き込むが、これは Markdown であり、既存の ESLint や TypeScript のフックは Markdown を対象としていない。したがって衝突は予期されないが、検証は未済である。

## 14. ロールアウトプラン

オーナーの意図は、採用を最大化することではなく、クリーンにリリースすることにある。以下のプランはその意図を反映している。

### フェーズ 1: ダークローンチ

本機能を出荷し、CI でデフォルト OFF の不変条件を検証する。テンプレートから派生したリポジトリに**最初のクローン時から同梱される**もの: `.claude/skills/growth/SKILL.md` の `/growth` Skill、`.claude/skills/quiet/SKILL.md` の `/quiet` コンパニオン Skill、`.claude/growth/preamble.md` の拡充コントラクト、`.claude/growth/notes/` 配下の 19 のプレシード済正典ドメインファイル（FR-005 と整合）。CLAUDE.md にはポインタブロックが含まれる。`scripts/check-growth-invariants.sh` が CI で実行され、3 つのチェック（Skill フラグ、エージェントのガード分岐、gitignore の方針）がすべてパスしていることを確認する。

学習者が最初に `/growth on` を実行したときに**実行時に作成される**のは `.claude/growth/config.json` のみである。追加の `.claude/growth/notes/<key>.md` ファイルは、学習者が `/growth domain new <key>` でカスタムドメインを開いたときにのみ作成される。

`.claude/growth/config.json` と `.claude/growth/notes/` はいずれもデフォルトで gitignore されている。同梱される `preamble.md` と seed 済ノートファイルはリポジトリから隠されてはいない（テンプレートの一部である）が、学習者が `notes/` 配下に蓄積した拡充はローカルにとどまる（学習者が gitignore エントリを反転させない限り）。

本機能は、Skill を実行した開発者のみが利用できる。ADR と PRD、リリースノート以上には言及しない。

### フェーズ 2: 公開利用可能

本機能を含むマイナーバージョンをリリースする。変更履歴では 1 行だけ本機能に言及する。ドキュメントはリポジトリに用意する。開発者は CLAUDE.md もしくは直接呼び出しを通じて Skill を発見したうえで利用する。ここではアナウンスもフィーチャーフラグもユーザーセグメントごとの段階的ロールアウトも行わない。これはテンプレートリポジトリであり、各派生プロジェクトは次のマイナーバージョンでテンプレートの参照を更新したときにメリットを得る。

フェーズ 2 のゴール: デフォルト OFF の回帰が実運用で顕在化していないこと、および拡充プロトコルが実プロジェクトで仕様どおりに動作することを確認する。

### フェーズ 3: 文書化

フェーズ 2 のシグナルがクリーンであること（デフォルト OFF の回帰がない／オープンクエスチョンから CRITICAL の未解決項目が出ていない／少なくとも 1 つの実プロジェクトが数週間にわたり Growth Mode を使っている）を確認したのちに、`docs/en/template-usage.md` に本機能を 3 つのレベルで説明する節を追加する。対応する日本語訳を `docs/ja/` に追加し、README.md でもやや目立つ形で言及する。

フェーズ 3 では本機能そのものの変更は不要である。アナウンスもドキュメント上での言及にとどめる。

いかなるフェーズにも、利用者側のマイグレーション作業は必要としない。既定の挙動は変わらず、テンプレートから派生した既存リポジトリは、開発者が明示的に `/growth on` を実行するまで影響を受けない。
