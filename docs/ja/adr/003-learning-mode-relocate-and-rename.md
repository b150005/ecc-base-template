# ADR-003: Growth Mode を Learning Mode に改名し、出力先を移動、遅延マテリアライズを導入する

## ステータス

採択済み。2026-04-24。

## メタデータ

- 日付: 2026-04-24
- 決定者: エージェントチーム (architect 主導、technical-writer 用語選定、product-manager 事例戦略、ui-ux-designer ディレクトリ設計)
- 置き換え対象: [ADR-001](001-developer-growth-mode.md) のディレクトリレイアウトおよびプレシーディングに関する決定（§Directory and seed files）、ADR-001・[ADR-002](002-growth-domains-location.md)・`.claude/growth/preamble.md`・全エージェントプロンプトにおける "notes" / "notebook" という用語全体
- 関連: [ADR-001](001-developer-growth-mode.md)、[ADR-002](002-growth-domains-location.md)、v2.0.0 破壊的リリース

## コンテキスト

Developer Growth Mode は v1.1.0 でリリースされ、v1.2.1 まで安定稼働してきました。2026 年 4 月にフォークした開発者からのフィードバックとエージェントチームによる監査を通じて、互いに切り離せない 3 つの問題が浮かび上がりました。それぞれが他の問題を増幅させる構造になっています。

1. **"notes" という語がアウトプットの価値を低く見せている。** 出力物は、多くのセッションを積み重ねて育てるドメイン別の体系的なリファレンスであり、手元のメモ帳よりも個人的な教科書に近いものです。"Notes" は非公式・一時的なキャプチャを連想させ、学習者（とエージェント）に対して、これらのファイルをその場限りの書き捨てメモとして扱うよう誘導してしまいます。積み重ねによって得た理解の蓄積であるという実態に沿っていません。

2. **ディレクトリが `.claude/` の中にあり、これはハーネス設定の領域である。** `.claude/` 配下に置かれている他のすべてのファイル（エージェント、スキル、設定、config）は、Claude Code が読み込む指示書です。学習者が積み上げていく知識ベースは**プロジェクトのアーティファクト**であって、ハーネスの機構ではありません。`preamble.md` や `config.json` と同じ場所に埋もれることで、まったく性格の異なる 2 種類のコンテンツが混在し、発見しにくく、チームメンバーと共有しにくく、選択的に gitignore することも難しくなります。

3. **19 個のプレースホルダーファイルをプレシードすることは、機能が何を学んだかについて嘘をついている。** フレッシュなクローンには 23 行のスタブがほぼ同一の内容で 19 個入っています。`ls .claude/growth/notes/` は、内容としてはまだ存在しない構造を示します。初回の拡充時の git diff には「プレースホルダー削除」のノイズが実際のコンテンツ追加のシグナルと混在します。preamble の Step 2 はすでに「ファイルが存在しない」ケースを正しく処理しているため、プレースホルダーは実質的な役割を果たしていません。

さらに独立した問題として、機能名である **Growth Mode** が抽象的すぎる点があります。この名称は、知識蓄積の柱（受動的な記録）も、v2.1.0 で設計中のコーチング柱（能動的なセッション内の行動変容）も、どちらも正確には表していません。両方の柱は**学習**の形態であるため、傘となる名称を改名すれば、両方をカバーできる名前になります。

本 ADR は、4 つの懸念点すべてを v2.0.0 の破壊的リリースとしてアトミックに解決するという決定を記録します。テンプレートはまだ若くフォーク数も少ない今、マイグレーションコストを受け入れます。

## 決定

### 1. 機能を改名する

| 表面 | v1.x | v2.0.0 |
|---|---|---|
| 機能名（長形式） | Developer Growth Mode | Developer Learning Mode |
| 機能名（短形式） | Growth Mode | Learning Mode |
| Skill コマンド | `/growth` | `/learn` |
| トレーラーラベル 1 | `## Growth: taught this session` | `## Learning: taught this session` |
| トレーラーラベル 2 | `## Growth: notebook diff` | `## Learning: knowledge diff` |

傘となる名称は **Learning Mode** になります。両方の柱（受動的な知識蓄積と、v2.1.0 での能動的コーチング）は、どちらも双方向の学習の形態です。エージェントが教え、学習者が学ぶ。この名称は意図的に双方向性を持ち、機能の全スコープをカバーします。

### 2. ディレクトリを `.claude/` の外へ移動する

機能ディレクトリ全体をリポジトリルートに移動します。

| 表面 | v1.x | v2.0.0 |
|---|---|---|
| 機能ディレクトリ | `.claude/growth/` | `learn/`（トップレベル） |
| Config ファイル | `.claude/growth/config.json` | `learn/config.json` |
| Preamble ファイル | `.claude/growth/preamble.md` | `learn/preamble.md` |
| 知識アウトプット | `.claude/growth/notes/<domain>.md` | `learn/knowledge/<domain>.md` |
| Skill ディレクトリ | `.claude/skills/growth/` | `.claude/skills/learn/` |
| ドキュメントディレクトリ | `docs/en/growth/` | `docs/en/learn/` |
| 事例ディレクトリ | (なし) | `docs/en/learn/examples/<domain>.md` |

`.claude/` にはハーネス config と Skills のみを残します——これはプラットフォーム固有の語彙です。学習者が所有するアーティファクトは、他のすべてのプロジェクトアーティファクトが置かれるプロジェクトルートに置かれます。Skill 自体は `.claude/skills/learn/` に残ります。Skills は Claude Code プラットフォームの構成物であって、プロジェクトアーティファクトではないためです。

### 3. 用語を置き換える

`notes` および `notebook` は全体を通じて `knowledge` に置き換えます。出力ディレクトリは `learn/knowledge/` です。トレーラー節は "knowledge diff" と表記します。エージェントプロンプト、preamble、ADR、ドキュメントのすべてが新しい用語を反映します。

### 4. knowledge ディレクトリを遅延マテリアライズする

v1.x は `.claude/growth/notes/*.md` に 19 個のプレースホルダーファイルを同梱していました。v2.0.0 では `learn/knowledge/` の下にファイルを **0 個** で出荷します。ディレクトリ自体は gitignore されます。あるドメインで初めて学びの機会が発生したとき、`learn/preamble.md` §7 に定義されたシード形状を使って `learn/knowledge/<domain>.md` が作成されます。これにより、`ls learn/knowledge/` は活動実績の正直なダッシュボードになります——ファイルは内容が積み上がったときにのみ存在します。

### 5. 別途読み取り専用の参考ファイルとして作業事例を提供する

> **実装は v2.1.0（PR #2）に延期されました。** 読み取り専用リファレンスとしての作業事例を `docs/en/learn/examples/<domain>.md` に配置するという決定は、ADR-003 の設計の一部として本文書に記録されています。19 個の Meridian に基づくファイルは、coaching pillar（ADR-004）と合わせて後続 PR（v2.1.0）で作成・同梱されます。v2.0.0 PR（#6）はすでに大規模で、事例は 48 エージェント時間程度の執筆が必要なためです。これらの存在に依存する挙動はありません — 遅延マテリアライズ不変式（§4）は単独で成立します。

学習者がポピュレートされた knowledge ファイルの見た目をすぐに把握できるよう、v2.1.0 では 19 個の作業事例を `docs/en/learn/examples/<domain>.md`（および日本語ミラーを `docs/ja/learn/examples/`）に同梱します。これらは以下の性質を持ちます。

- **読み取り専用のリファレンス。** エージェントは `docs/en/learn/examples/` の下を読んだり引用したり書き込んだりしません。preamble の読み書き対象は、このツリーを明示的に除外しています（§8）。
- **共通の架空プロジェクトに基づいている。** 19 個の事例すべてが *Meridian*（B2B タスク管理 SaaS：Go + Gin + PostgreSQL + Redis バックエンド、React + TanStack Query フロントエンド、Kubernetes + GitHub Actions デプロイ）を参照します。ドメインをまたいだ参照が自然に感じられるよう設計されています。
- **レベルを意識している。** 各コンセプトエントリには、ADR-001 の 3 段階推論ラダーに合わせて `[JUNIOR]`、`[MID]`、`[SENIOR]` のマークが付いています。
- **ライブコンテンツと明確に区別されている。** すべての事例ファイルは、それが読み取り専用のリファレンスであってユーザーが生成したファイルではないことを明示するバナーで始まります。

事例は遅延マテリアライズとの共存を意図しています。v2.1.0 が出荷された後は、フレッシュなフォークは `learn/knowledge/` が空で、`docs/en/learn/examples/` に 19 個のポピュレートされたファイルがある状態になります。学習者は事例を読んでフォーマットを理解し、学びの機会が発生するにつれて自分自身の知識ベースを育てていきます。v2.1.0 出荷前（すなわち v2.0.0 と v2.1.0 の間）は `docs/en/learn/examples/` ツリーは存在せず、学習者は preamble と解説のみを参照することになります。

## 意思決定の軸

この決定は、以下の優先順位で最適化されています。

1. **デフォルト OFF のバイト同一不変式を維持する**（preamble §1）。すべての変更において、Learning Mode が無効のときにバイト同一の出力が保証され続けなければなりません。
2. **`.claude/` のハーネス設定とプロジェクトアーティファクトの境界を尊重する。** プロジェクトアーティファクトはプロジェクトルートに置く。ハーネス config と Skills は `.claude/` 配下に置く。
3. **用語をアーティファクトの実際の形態に合わせる。** ドメイン別に整理され、積み上がり、相互参照されるリファレンスは *knowledge* であって *notes* ではない。
4. **フレッシュなクローンの状態を正直に保つ** — 内容が積み上がるまでディスク上にファイルは存在しない。
5. **決定論的な CI 強制を維持する。** 不変式チェックに LLM を介在させない。すべてを grep 可能な状態に保つ。
6. **フォーカーのマイグレーション負担を最小化する** — テンプレートはまだ若く、多くのフォークは機能を有効化していない。マイグレーションコストは少数の対象にとどまる。

## 結果

### ポジティブな結果

- **ハーネス/アーティファクトの境界がクリアになる。** `.claude/` は Claude Code プラットフォーム領域としてのアイデンティティを取り戻します。`learn/` は明確にプロジェクトアーティファクトです。
- **フレッシュなクローンの状態が正直になる。** `ls learn/knowledge/` は実際のコンテンツが存在するまで何も返しません。初回拡充時の git diff は実際のコンテンツのみを示し、プレースホルダーノイズが混在しません。
- **傘となる名称が改善される。** "Learning Mode" は既存の知識蓄積柱と v2.1.0 のコーチング柱の両方を無理なくカバーします。
- **語彙が統一される。** "Knowledge" はアーティファクトの長期的なリファレンスとしての形態に合致します。トレーラーラベルの "knowledge diff" も自然に読めます。
- **事例がポピュレートされた状態で提供される。** フォーカーは、ライブ学習者サーフェスを汚染することなく、15〜20 セッション後の自分の `learn/knowledge/` がどのように見えるかのリアルなイメージを得られます。

### ネガティブな結果

- **既存のフォークにとって破壊的変更となる。** Growth Mode を有効化し `.claude/growth/notes/` にコンテンツを蓄積したフォークは、パスをマイグレーションする必要があります。`docs/en/migration/v1-to-v2.md` の散文マイグレーションガイドが手順（`git mv`、検索置換）をカバーします。マイグレーションスクリプトは提供しません——テンプレートのフォーク数は少なく、操作は機械的です。
- **一回限りの大規模 PR。** Growth Mode に言及するリポジトリ内のすべてのドキュメントが、単一の v2.0.0 PR で触れられます。レビュー負担は通常の機能 PR より高くなりますが、多くの編集は機械的なパス/用語の置換であることが補填材となります。

### ニュートラルな結果

- **ADR-001 の本文は書き直さない。** 冒頭に "Superseded in part by ADR-003"（ADR-003 により部分的に supersede された）というヘッダーノートを付けます。アーキテクチャの実体（レベル体系、enrichment 契約、trailer）は引き続き ADR-001 が管理します。supersede されるのはディレクトリレイアウトと用語のみです。
- **ADR-002 は用語パスを受ける。** エージェントファイルの `## Growth Domains` セクションマーカーは、傘となる名称に合わせて `## Learning Domains` に改名されます。ADR-002 の実体（プロンプト本文での宣言、CI 再アンカー）は変更されません。
- **CI 不変式スクリプトの改名。** `scripts/check-learn-invariants.sh` に改名され、grep ターゲットを更新します（`.claude/growth/notes/` → `learn/knowledge/`、`## Growth Domains` → `## Learning Domains`）。3 つの決定論的チェックはそのまま維持されます。

### 作業事例の成果物（v2.1.0 / PR #2 に延期）

### ADR-003 のスコープ外

- **コーチング柱。** Output Styles 互換のセッション内行動モード（`hints`、`socratic`、`pair`、`review-only`、`silent`）は ADR-004 および PR #2（v2.1.0）に延期されます。現在の決定は、2 度目の破壊的変更なしに `learn/config.json` に `coach` サブツリーを追加できる余地を残しています。
- **分類体系や enrichment 契約の構造的変更。** それらは引き続き ADR-001 が管理します。ADR-003 は移動と改名を行うものであり、プロトコルそのものを再構築するものではありません。
- **自動化されたマイグレーションスクリプト。** メンテナーはスクリプト提供ではなく散文のみのマイグレーション手順を明示的に選択しました。テンプレートのフォーク数は少なく、操作は機械的です。
