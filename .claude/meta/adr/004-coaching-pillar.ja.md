# ADR-004: Learning Mode コーチングピラー（Output Styles 互換の行動モード）

## ステータス

採択済み。2026-04-25。

## メタデータ

- 日付: 2026-04-25
- 決定者: エージェントチーム (architect 主導、implementer は強制サーフェス担当、technical-writer はスタイルファイル台帳担当、ui-ux-designer はコマンドサーフェスのエルゴノミクス担当)
- 関連: [ADR-001](001-developer-growth-mode.md)（本 ADR が補完する知識ピラー）、[ADR-003](003-learning-mode-relocate-and-rename.md) §G「コーチングピラー」（v2.0.0 から v2.1.0 に明示的に延期された項目 — 本 ADR がその実装を提供する）、[docs/en/prd/developer-learning-mode.md](../prd/developer-learning-mode.md)
- 対象リリース: v2.1.0。ADR-003 §5 から延期された Meridian 作業事例の成果物は、レビュー負荷上の理由から本来の PR #2 スコープを分割し、v2.2.0 で別途出荷します。

## コンテキスト

Learning Mode は v2.0.0 で、**知識蓄積**という単一のピラーとともに出荷されました。学びの機会が生じるたびに、各エージェントは永続的なエントリを `learn/knowledge/<domain>.md` に書き込みます。出力物は、多くのセッションを重ねながら育てていくドメイン別の体系的なリファレンスです。このピラーは**後発かつ受動的**なものです — エージェントは通常のアーティファクトを生成した上で、教えた内容を記録します。学習者はアーティファクトを通常どおり読み、その後に knowledge diff を参照します。

既存の仕組みでは対応できないニーズが 2 つあります。

1. **手を動かして実装したい学習者**。知識ピラーはエージェントが生成するものを変えません — 生成された結果にアノテーションを付けるだけです。「ログインエンドポイントは自分で実装したい。コードは書かないで次のステップだけ教えて」という学習者には、その希望を伝えるための手段がありません。現状では散文的な指示でエージェントに頼むしかなく、次のターンで守られるかどうかはターンごとに異なります。

2. **完成した答えではなく、判断の形を手伝ってほしい学習者**。「このキャッシュは使うべきか？」という問いに、完成したキャッシュの実装が返ってきた場合、判断のプロセスが丸ごとスキップされます。知識ピラーがあとでトレードオフのメモを書いてくれたとしても、判断はすでに学習者の代わりに下されています。

どちらのギャップも、**セッション内の行動変容**を必要とします — エージェントが実装の過程で異なる振る舞いをする必要があり、単に後から記録するだけでは不十分です。これは知識蓄積とは別の軸です。知識は記録されるものであり、コーチングはエージェントの動き方です。

Claude Code の **Output Styles** 機能は、最も近いプラットフォームの基本機能です。Output Styles は、システムプロンプトのコーディングスタイルセクションを、Markdown ファイルから読み込んだ別の命令セットで置き換えます。「このセッションは学習チューターモードで動いてほしい」というような、セッション全体の挙動変更に向いています。ただし、Output Styles には Learning Mode にそのまま合わない 2 つの特性があります。

- Output Styles は**セッション開始時に束縛され**、コーディング命令を全体的に置き換えます。Learning Mode では、コーチングを通常の動作に重ねて適用でき、セッションを再起動せずにスイッチできる必要があります。
- Output Styles は、Learning Mode のデフォルト OFF の不変条件、知識ピラーの状態、モデル呼び出し禁止の保証（`disable-model-invocation: true`）を認識しません。直接採用すれば、機能がすでに担保している不変条件からコーチングが逸脱してしまいます。

コーチングピラーは Output Styles の**ファイルフォーマット**を採用します — 移植可能で標準に準拠しており、Claude Code エコシステムをすでに知っている人が読めます — ただし、状態とディスパッチは Learning Mode の既存サーフェスの内側に保持します。本 ADR はそのアーキテクチャと 6 つのコーチスタイル（不活性な `default` と、5 つの能動的スタイル）を記録します。

## 決定

コーチングピラーを、知識ピラーと直交する軸として出荷します。両ピラーはそれぞれ OFF、個別に ON、あるいは同時に ON にすることができます。コーチングピラーは、`.claude/skills/learn/coach-styles/` 配下の Output Styles 互換 Markdown ファイルとして表現される 6 つのスタイル（不活性な `default` と 5 つの能動的な行動モード）を導入します。アクティブなスタイルは `learn/config.json` に書き込まれ、知識ピラーを強制するのと同じエージェントのガード分岐によって強制されます。

### 1. 決定論的な行動ルールを持つ 6 つのコーチスタイル

| スタイル | 行動ルール | 使いどころ |
|---|---|---|
| `default` | エージェントは通常どおり動作する。保留も追加の教示もなし。コーチ OFF と同等。 | このセッションではコーチは不要。 |
| `hints` | エージェントは次の具体的なステップを特定し、関連するパターンや API の名前を挙げ、対象関数の本体を書く前に止まる。ステップ、パターン名、一行の根拠を含む `## Coach: hint` ブロックを出力する。足場（インポート、シグネチャ、テストスタブ）は書いてよい。対象関数の本体は書かない。 | 実装コードを自分で書きたい学習者向け。 |
| `socratic` | how または why の質問に対し、答えると設計が一つに絞れるような、焦点の定まった問いを一つだけ返す。問いを出したターンにコードは書かない。学習者が答えた後は通常の行動に戻る。 | 学習者に文脈があれば自力で選択できる設計判断に適している。 |
| `pair` | 判断が必要な箇所に `// TODO(human): <一行の指示>` マーカーを付けた完全な足場を書く。マーカーは変更行の約 30% を上限とし、足場は本当に骨格であることを保証する。テストは完全に書くので、学習者はそれを通過させることを目標にできる。 | 構造を渡してもらいながらアルゴリズム自体を担当したい学習者向け。 |
| `review-only` | 本番コードの実装を拒否する。コードを読み、テストを実行し、学習者が提出または既にディスク上にあるコードのレビューを構造的に行う。明示的に求められた場合はテストを書いてよい。 | 学習者が主導しており、エージェントに著者ではなくレビュアーとしての役割を求めるとき。 |
| `silent` | エージェントは通常どおり動作し、かつ、このスタイルが有効な間すべての `## Learning:` および `## Coach:` 末尾節を抑制する。教示モードの逆。 | 学習者がフロー状態にあり、応答に教育的ノイズを入れたくないとき。 |

スタイルは排他的です — 常に 1 つだけがアクティブです。スタイルの切り替えは、次のエージェントターンから有効になります。

### 2. ハイブリッドアーキテクチャ（Output Styles ファイルフォーマット + Learning Mode config の状態）

スタイルファイルは Output Styles 互換の Markdown で記述します。YAML フロントマターとボディからなります。フロントマターには `name`、`description`、そして Learning Mode 拡張フィールドである `behavior-rule:` が含まれ、エージェントがターン時に強制する決定論的ルールをエンコードします。ボディはファイルを読む人間向けの散文です。

```yaml
---
name: hints
description: Name the next step and the pattern; stop before the function body.
behavior-rule: >
  Identify the next concrete step toward the learner's goal. Name the relevant
  pattern or API. Write scaffolding (imports, signatures, test stubs) only.
  Do not write the body of the target function. Emit a `## Coach: hint` block
  with step, pattern, and one-line rationale.
stop-markers:
  - "<!-- coach:hints stop -->"
---
```

**どの**スタイルがアクティブかの Source of Truth は `learn/config.json` であり、セッション開始時の束縛ではありません。これにより、`/learn coach <style>` が再起動なしにセッション途中でスイッチでき、ガード分岐が 1 つの JSON ファイルを読むだけですべての Learning Mode の状態を解決でき、`disable-model-invocation: true` の著者境界も損なわれません。

スタイルファイルは `.claude/skills/learn/coach-styles/<style>.md` に配置されます。Skill 本体はこのディレクトリを一覧することで利用可能なスタイルを発見します — コードを変更せずにファイルを追加するだけで新しいスタイルを足せます。

### 3. Config スキーマの拡張

```json
{
  "enabled": true,
  "level": "junior",
  "focus_domains": [],
  "coach": {
    "style": "default",
    "trailers": "auto",
    "scope": "session"
  },
  "updatedAt": "2026-04-25T00:00:00Z"
}
```

- `coach.style` — `default | hints | socratic | pair | review-only | silent` のいずれか。値が欠如しているか解析不能な場合は `"default"` に解決される。`coach-styles/` 配下のいずれのファイルとも一致しないスタイル名は、次回の `/learn status` で警告を出した上で `"default"` に解決される。
- `coach.trailers` — `auto | always | never`。`auto` では、`silent` スタイルがトレーラーを抑制し、他のスタイルでは知識ピラーが ON のときにトレーラーを出力する。
- `coach.scope` — `session | persistent`。`session` では、Skill は新しいセッションの最初の呼び出し時に `coach` サブツリーをリセットし、`coach.style = "default"` に書き戻す（`scope = session` の場合）。`persistent` では、スタイルはセッションをまたいで持続する。
- **後方互換性**: `coach` キーのない v2.0.0 の config は `coach.style = "default"` に解決される。既存インストールで挙動は変わらない。デフォルト OFF の不変条件は保たれる。

### 4. ピラーの合成ルール

2 つのピラーは直交します。

- **知識 OFF、コーチ OFF** — デフォルト状態。Learning Mode が全くない場合とバイト単位で同一。
- **知識 ON、コーチ OFF（= `default`）** — v2.0.0 の挙動。
- **知識 OFF、コーチ ON** — 許可される。ノートのオーバーヘッドなしに能動的なコーチングを受けられる。
- **知識 ON、コーチ ON** — 両方のレイヤーが重なる。

相互作用のルール:

- `socratic` では、エージェントの問いかけ自体が学びの機会になり得る。問いが重要な概念を明らかにする場合、エージェントは問いかけをする同じ応答の中で、拡充プロトコルにしたがって `learn/knowledge/` に書き込む。
- `silent` では、知識ピラーが ON のとき、知識の書き込みは引き続き行われます。抑制されるのはチャットに見えるトレーラーのみです。`/learn status` は常に直近 N 件の knowledge diff を報告するため、silent による書き込みが見えなくなることはありません。
- `level: junior` は自動的に `hints` やその他のコーチスタイルに結びつきません。すべてのレベルでデフォルトのコーチは `"default"` です。レベルは説明の角度を制御し、コーチはエージェントの作業の形を制御します。この 2 つの軸は互いに制約しません。

### 5. Skill コマンドサーフェスの追加

`/learn` Skill に `coach` サブコマンドグループが追加されます。

```
/learn coach <style>                      — スタイルを設定する（/learn coach style <style> と等価）
/learn coach off                          — /learn coach default と等価
/learn coach list                         — 発見したスタイルファイルとその説明を一覧する
/learn coach show <style>                 — 指定スタイルの行動ルールを表示する
/learn coach scope <session|persistent>   — coach サブツリーの永続スコープを設定する
```

Skill の `disable-model-invocation: true` フラグはすべての coach サブコマンドに適用されます。エージェントのターンでは、コーチスタイルを切り替えたり無効にしたりする操作は一切できません — スタイルを選択できるのは学習者のみです。

## 検討した代替案

| 代替案 | 長所 | 短所 | 採用しない理由 |
|---|---|---|---|
| **A. Claude Code Output Styles の直接採用**。学習者が `/output-style learning-tutor` を呼び出し、Learning Mode がコーチ状態として Output Styles API を読む。 | プロトコルの発明が不要。プラットフォームがすでにスタイルファイルの読み込みを知っている。保守が必要なサーフェスが最小限。 | セッション開始時に束縛される — セッション途中でスタイルを切り替えるには再起動が必要で、文脈が失われる。Output Styles はコーディング命令セクションを重ねるのではなく丸ごと置き換える。Learning Mode の不変条件（`disable-model-invocation`、デフォルト OFF のガード、ピラーの合成）を認識しない。学習者は 1 つの機能を制御するために 2 つのトグル（`/output-style` と `/learn`）を理解する必要がある。 | 採用しない。知識ピラーですでに強制している不変条件が機能の境界から逸脱する。 |
| **B. 完全な独自実装**。カスタムファイルフォーマット、カスタム発見パス、Output Styles との参照関係なし。 | 完全なコントロール。進化するプラットフォーム機能への依存がない。短期的にはシンプルになり得る。 | 保守すべき仕様が増える。スタイルファイルが他の Output Styles 利用者へのポータビリティを失う。Output Styles をすでに知っているフォーカーが独自バリアントを習得しなければならない。既存の Output Styles ファイルを出発点として利用できなくなる。 | 採用しない。メリットなしに優れたファイルフォーマットを再発明することになる。 |
| **C. ハイブリッド — Output Styles ファイルフォーマット、Learning Mode の状態とディスパッチ（採用）**。 | 移植可能なファイルフォーマット。Output Styles を知っている人が読める。状態が `learn/config.json` にあるため不変条件が保たれる。セッション途中で切り替え可能。新スタイルはファイルを追加するだけで、コード変更不要。 | `behavior-rule:` フロントマター拡張とそのセマンティクスを文書化する一度限りのコスト。 | 採用。実績あるファイルフォーマットを借用しながら、すべての Learning Mode の不変条件を保全する。 |

## 意思決定の軸

優先順位の高い順に:

1. **デフォルト OFF のバイト同一不変式**。config に `coach` キーのない v2.1.0 インストールは v2.0.0 とバイト単位で同一の出力を生成しなければならない。コーチブランチのガードは知識ブランチのガードを踏襲する。
2. **ハーネス/アーティファクトの境界**。スタイルファイルはプラットフォーム隣接の機構であるため `.claude/skills/learn/coach-styles/` 配下に置く。学習者が蓄積した知識は `learn/knowledge/` に置く。ADR-003 で確立した境界を尊重する。
3. **決定論的な強制**。行動ルールはスタイルファイルに命令形で記述し、ターン時に確認する。ストップマーカー（`<!-- coach:hints stop -->`）により、スタイルが実行されたことを grep で検証できる。LLM を介在させた不変条件チェックは行わない。
4. **モデルによる状態変更の禁止**。`disable-model-invocation: true` はすべての `coach` サブコマンドに適用される。スタイルを選択するのは学習者であり、モデルではない。
5. **知識ピラーとの直交合成**。コーチングピラーは知識ピラーが書くものを変えない。知識ピラーはアクティブなコーチスタイルを変えない。それぞれがもう一方を触れることなくトグルできる。
6. **セッション途中での切り替え可能性**。学習者は作業の形が変わるにつれて、1 セッション中に `hints` と `default` を何度でも切り替えられる。これが可能なのは、状態がセッション開始時の束縛ではなく `config.json` に置かれているためだ。

## 結果

### ポジティブな結果

- 能動的なコーチングを求める学習者は、ファイルで裏付けられた行動ルールを持つ決定論的な行動セットから選択できます。ルールはレビューもフォークも可能です。
- スタイルファイルは移植可能です。カスタムのコーチスタイルを望むチームは、ファイルを `coach-styles/` に追加して名前で参照するだけです — コード変更も Skill の書き直しも不要です。
- 著者境界は変わりません。モデルは学習者をペアプログラミングモードに昇格させたり、レビューのみモードに降格させたりすることができません。できるのは学習者だけです。
- `silent` スタイルにより、学習者は知識ピラーを無効にすることなくトレーラーのノイズから逃れる出口が得られます。蓄積を維持しながらチャット応答を静かにできます。
- ハイブリッドフォーマットにより、Claude Code が後のリリースでプラットフォームの Output Styles レジストリにスタイルを登録する仕組みを持った場合、このテンプレートのスタイルをそこに移植できます — ファイルはすでに正しい形をしています。

### ネガティブな結果

- プレアンブルが約 50 行増加します（新しい §§15–20）。これは一度限りのコストであり、新しい節は独立しています。
- スタイルごとの決定論的な強制は、命令的なルールに対するモデルの遵守に依存します。ストップマーカーと権限の剥奪（たとえば、`review-only` で実行されるターンではエージェントの write ツール権限が保留されます）がドリフトを軽減しますが、コーチングピラーは、行動の正確さが grep 可能な不変条件だけでなく規律の問題でもある初めての箇所です。スタイルファイルの PR レビューが補完的なコントロールとなります。
- `silent` + 知識 ON では、チャットに diff トレーラーが表示されることなく知識ファイルが変更されます。書き込みを予期していない学習者には見えにくいと感じる可能性があります。緩和策: `/learn status` は常に直近 N 件の knowledge diff を報告し、トレーラー抑制の挙動はスタイルのボディに明示的に文書化されています。

### ニュートラルな結果

- 既存の v2.0.0 インストールは透過的にアップグレードされます。`coach` キーのない config ファイルは `coach.style = "default"` に解決され、何も変わりません。
- スタイルファイルの日本語訳は本 ADR のスコープ外です。スタイルファイルのボディは出荷時に英語であり、行動ルールは学習者の言語に関わらず適用されます。
- `/quiet` Skill（ADR-001 でワンショットのトレーラー抑制のために導入）は、`coach: silent` とは分離したまま残ります。`/quiet` はシングルターンの抑制であり、`silent` は変更されるまで持続するスタイルです。どちらも存在するのは、異なる著者境界に対応するからです。

## 実装上の注記

フェーズ別にまとめます。実装者エージェントは ADR-004 が決定されたのちにこれを引き継ぎます。

### フェーズ 1: Config スキーマとガード

- `learn/config.json` スキーマを、オプションの `coach` サブツリーを受け入れるよう拡張する。既存の config が `/learn` をアップグレード後初めて呼び出したとき、`coach: { style: "default", trailers: "auto", scope: "session" }` を追加するマイグレーションステップを書く。既存のフィールドは保持する。
- すべての Learning 対応エージェントのガード分岐を拡張する: `learn/config.json` を読み込んだあと、`coach.style` が `default` 以外の値であり、かつスタイルファイルが `.claude/skills/learn/coach-styles/` 配下に存在する場合、スタイルファイルを読み込み、このターンの行動ルールを適用する。`coach.style` が欠如、不正、またはファイルが存在しない場合は `default` にフォールバックする。

### フェーズ 2: スタイルファイル

- `.claude/skills/learn/coach-styles/` を作成し、スタイルごとに 1 ファイルを置く: `default.md`、`hints.md`、`socratic.md`、`pair.md`、`review-only.md`、`silent.md`。各ファイルは決定事項 §2 に示したフロントマター形式を使い、人間のレビュアー向けに行動を説明する散文のボディを持つ。
- 各スタイルファイルの具体的な内容は実装者の成果物です。本 ADR は形式と行動ルールを規定しますが、ファイルの本文を著述しません。

### フェーズ 3: プレアンブルの追加

`learn/preamble.md` に §§15–20 を追加します:

- **§15 コーチングピラー概要** — 概要、デフォルト OFF の不変条件をコーチ向けに再述、知識ピラーとの直交性。
- **§16 コーチスタイルの解決順序** — `config.coach.style` → スタイルファイル → 組み込みの `default` へのフォールバックという読み取り順序。
- **§17 スタイルごとの行動コントラクト** — 決定事項 §1 から得られる 6 つの決定論的ルール（`default` と 5 つの能動的スタイル）を、プレアンブルの命令形スタイルで記述。
- **§18 コーチ × 知識の合成** — コーチの行動（特に `socratic`）が知識の対象となる瞬間を生み出すとき、および `silent` ルールのトレーラー抑制について。
- **§19 コーチトレーラーのフォーマット** — `## Coach: <style> output` 見出しのフォーマット、`silent` と `coach.trailers: never` での抑制ルール。
- **§20 スタイルファイルのフォーマット** — フロントマタースキーマ（`name`、`description`、`behavior-rule`、`stop-markers`）、Output Styles に対する拡張、スタイルファイルがセッション開始時ではなくガード時に読み込まれるというルール。

## スコープ外

- **スタイルファイルの具体的な内容**。本 ADR は行動ルールを示しますが、各 `<style>.md` ファイルのボディを著述するのは PR #2 での実装者のタスクです。
- **スタイルファイルの日本語翻訳**。行動ルールは言語に依存しません。ボディの散文は出荷時に英語です。後のリリースで本 ADR を変更せずにボディを翻訳できます。
- **最初の 6 スタイル以外の新しいコーチスタイル**。追加のスタイル（例: `drill`、`refactor-coach`）は v2.1.0 以降の懸念事項です。ドロップインファイルフォーマットにより、ここで述べた不変条件を変更しない限り、新 ADR なしで追加できます。
- **Claude Code の Output Styles レジストリとの統合**。プラットフォームがレイヤードスタイルのサポートを追加した場合、将来の ADR で登録ステップを記録します。それまでは、スタイルファイルはプラットフォームではなく Learning Mode 独自のガードによって読み込まれます。
- **知識ピラーの内部ルールに関するあらゆる事項**。それらは ADR-001 と v2.0.0 プレアンブルが管理し続けます。ADR-004 は 2 番目のピラーを追加しますが、最初のピラーを変更しません。
