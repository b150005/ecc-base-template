## 「依存関係を追加する」判断ツリー  [JUNIOR]

### 第一原理からの説明  [JUNIOR]

プロジェクトに追加されるすべての依存関係は長期的なコミットメントです。その依存関係は、プロジェクトが書いたわけでなく、更新のたびに完全に監査できるわけでもなく、依存関係の作者が進化させていく中で互換性を保ちながらメンテナンスし続けなければならないコードをもたらします。「このパッケージを追加すべきか?」という問いに対する答えは、「それは問題を解決するか?」を問うことで得られません。ほぼすべてのパッケージは謳われた問題を解決します。問うべきは、時間をかけてその依存関係を所有するコストが代替案のコストより低いかどうかです。

依存関係の候補に対して 3 つの代替案が存在します:

1. **標準ライブラリを使う。** ほとんどの言語は一般的な操作の実装を同梱しています: HTTP クライアント、JSON パース、ソート、暗号プリミティブ、時刻のフォーマット。標準ライブラリのコードは言語の作者がメンテナンスし、主要な言語バージョンでのみ変更され、外部のサプライチェーン面を持ちません。

2. **実績のあるサードパーティパッケージを使う。** 標準ライブラリに機能がない場合、またはコミュニティのパッケージがより優れたインターフェース（落とし穴が少ない、エラーメッセージが豊富、テストカバレッジが広い）を提供する場合、確立されたパッケージが正しい答えである場合があります。「実績がある」とは: 広く使用されており、積極的にメンテナンスされており、明確なリリースリズムとパブリックな Issue トラッカーを持つことを意味します。

3. **自分で書く。** 問題が小さく、インターフェース要件が特殊で、大幅なラッピングなしに既存のパッケージがフィットしない場合、実装を自分で書くことで依存関係の表面積が減り、インターフェースをプロジェクト自身のコントロール下に置けます。

Meridian がすべての候補に適用する判断ツリー:

```
標準ライブラリできれいに解決できるか?
  Yes → 標準ライブラリを使う。終了。
  No  →
    実装を正しく行うことは非自明か?
    (暗号、プロトコルパース、分散協調)
      Yes → メンテナーが複数いて直近 12 ヶ月にリリースがある
             実績のあるパッケージがあるか?
             Yes → ピニングポリシーで依存関係を追加する（下記参照）。
             No  → 再評価する; このカテゴリのシングルメンテナーパッケージは
                   サプライチェーンリスク。
      No  → 実装が 100 行未満で明確に境界が定まるか?
             Yes → 自分で書く。依存関係の表面積ゼロ。
             No  → ベンダーリスクチェックリストでパッケージを評価する。
```

「実装を正しく行うことは非自明か」という基準が最も難しい判断です。暗号操作はこのカテゴリに普遍的に入ります: 正しい答えは常にメンテナンスされたライブラリ（`golang.org/x/crypto`）であり、手作りは決してしません。HTTP クライアントの設定はこのカテゴリに入りません: Go の `net/http` は十分な機能を持っており、標準ライブラリへの薄いラッパーがほぼ常に十分です。

### 慣用的な応用  [MID]

Meridian の Go バックエンドでは、標準ライブラリの境界は HTTP クライアントと標準的な JSON マーシャリングに引かれています。プロジェクトは Slack SDK や Webhook コンシューマーへのアウトバウンド HTTP 呼び出しに `net/http` を直接使用しています。サードパーティの HTTP クライアントライブラリは追加しません。

TypeScript フロントエンドでは、境界はデータフェッチに引かれています。カスタムの再試行とキャッシュロジックを持つ `fetch` を使う代わりに、Meridian は TanStack Query（`@tanstack/query`）を使用しています。これは「標準ライブラリを使う」からの意図的な例外です。この決定は、TanStack Query の stale-while-revalidate セマンティクス、キャッシュの無効化、バックグラウンドリフェッチの動作が手作りで正しく実装するには約 600〜800 行必要であるという事実に基づいています。その行数と関連するエッジケース（並行フェッチでのレースコンディション、アンマウント時のメモリリーク防止）が、判断ツリーにおいてこれを「非自明」な実装にします。

### トレードオフと制約  [SENIOR]

判断ツリーは、`go.mod` や `package.json` にパッケージが追加される前に、明示的なコストとベネフィットの言語化を強制します。ツリーのコストは「パッケージを見つけて追加する」よりも時間がかかることです。ベネフィットは依存関係の肥大化を防げることです — 小さな問題をそれぞれ解決する数十のパッケージが各自のリリースリズム、セキュリティアドバイザリ面、廃止の可能性を持つコードベース。

Meridian の Go の `go.mod` は 2 年間の開発後に直接依存関係が 14 個です。このフィルタを適用しなかった同等のプロジェクトは 40〜60 個持っているかもしれません。直接依存関係が 1 つ増えるごとに推移的依存関係が追加され、それぞれが潜在的なサプライチェーンリスクです（下記の推移的依存関係の監査を参照）。追加時のフィルタコストは評価に約 10 分です; 削除された各依存関係の継続コストはゼロです。

判断ツリーの唯一の失敗モード: 標準ライブラリパスへの誤った信頼感を生む可能性があります。Go の `net/http` は有能ですが、再試行とバックオフを持つアウトバウンド呼び出しには冗長です。プロジェクトはその冗長さを受け入れてきました; それが過度だと判断する将来のセッションでは、狭い HTTP クライアントライブラリが正当化されるかどうかを再検討すべきです。

## バージョンピニングポリシー  [MID]

### 第一原理からの説明  [JUNIOR]

依存関係のバージョン制約は、インストール時にどのバージョンのパッケージが許容されるかをパッケージマネージャーに伝えます。制約は 2 つの形をとります:

- **正確なピン**: バージョン `1.4.2` を正確にインストールする。利用可能であっても新しいパッチやマイナーリリースは使用されない。インストールされるバージョンは完全に決定的です。
- **範囲制約**: npm の semver セマンティクスで `^1.4.0`（1.4.x から 1.x.x と互換性がある）を満たす最新バージョンをインストールする。インストールされるバージョンは、新しい互換性のあるリリースが公開されると変わる場合があります。

この差異が重要なのは、範囲制約では、異なる日にリポジトリをクローンした 2 人の開発者が同じパッケージの異なるバージョンをインストールする可能性があるためです。一方は `1.4.3` を持ち、もう一方は `1.5.1` を持つかもしれません。`1.5.1` がリグレッションを導入した場合、開発環境は診断が難しい方法で異なります。

ロックファイル（`go.sum`、`package-lock.json`、`pnpm-lock.yaml`）は、インストール時の正確に解決されたバージョン — とその暗号ハッシュ — を記録することでこの問題を解決します。以降のすべてのインストールはロックファイルから読み取り、制約が何を言おうとも同一に解決します。ロックファイルが実際のピニングメカニズムです; `package.json` や `go.mod` のバージョン制約は意図の宣言であり、強制の表面ではありません。

### 慣用的な応用  [MID]

**Go 側。** Go モジュールは設計上正確です。`go.mod` は最小バージョンを記録し、`go.sum` は解決されたモジュールツリーの暗号ハッシュを記録します。`go get` と `go mod tidy` はどちらのファイルも明示的に更新します。これらの操作の間に暗黙的なアップグレードは発生しません。Meridian の結果的なポリシーはシンプルです: `go.sum` をソース管理にコミットし、gitignore に追加せず、PR で説明のない `go.sum` の変更をコードレビューのフラグとして扱います。

**npm/TypeScript 側。** `package.json` は慣例としてアプリケーション依存関係にキャレット範囲（`^`）を使用しますが、これは決定性を強制するためにロックファイルを必要とします:

```json
// package.json — ロックファイル強制を持つキャレット範囲
{
  "dependencies": {
    "@tanstack/react-query": "^5.28.0",
    "react": "^18.3.1",
    "react-dom": "^18.3.1"
  },
  "devDependencies": {
    "typescript": "^5.4.5",
    "vite": "^5.2.8"
  }
}
```

ロックファイル（`pnpm-lock.yaml`）が正確に解決されたバージョンを記録します。Meridian の CI は `pnpm install --frozen-lockfile` を実行し、ロックファイルが `package.json` と同期していない場合はインストールを拒否します。これによりロックファイルが強制の表面となり、キャレット範囲ではありません。

キャレット範囲は意図を伝えるために保持されています: 「私たちはマイナーラインを追跡する」。開発者が `pnpm update @tanstack/react-query` を実行して新しいマイナーを取り込む場合、それは期待される動作です; 更新されたロックファイルは CI グリーンのゲートを持つ専用の更新 PR でコミットされます。自動パッチ更新は Dependabot で処理されます（下記のこれまでの理解の節を参照）。

**GitHub Actions の正確なピニング。** CI ワークフローファイルはサードパーティのアクションをバージョンタグではなくコミット SHA にピン留めします:

```yaml
# .github/workflows/ci.yml
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
- uses: actions/setup-go@d35c59abb061a4a6fb18e82ac0862c26744d6ab5  # v5.5.0
```

タグベースのピニング（`actions/checkout@v4`）はサプライチェーンセキュリティには不十分です: タグはミュータブルです。侵害されたタグは、ワークフローファイルを変更せずに悪意のあるコミットを指すように移動できます。コミット SHA はイミュータブルです; ピン留めされたハッシュは評価時に実行された正確なコードに暗号的に結びついています。人間が読めるバージョンコメント（`# v4.2.2`）はミュータブルな参照を導入することなく読みやすさを維持します。

### トレードオフと制約  [SENIOR]

GitHub Actions の正確な SHA ピニングは、上流のアクションがリリースされると Dependabot が SHA とバージョンコメントの両方を更新する PR を開くことを意味します。コストはレビューする Dependabot PR クラスが 1 つ増えることです; ベネフィットは悪意のあるタグの移動が可視の PR なしに CI で任意のコードを実行できないことです。

キャレット + ロックファイル戦略には 1 つのローカル環境リスクがあります: `--frozen-lockfile` なしに `pnpm install` を実行した開発者が、新しいマイナーを静かに取り込んでロックファイルの変更をコミットする可能性があります。Meridian は `pnpm install --frozen-lockfile` を実行し、ロックファイルが変更される場合は失敗するプリコミットフックでこれを防いでいます。

## 推移的依存関係の監査  [MID]

### 第一原理からの説明  [JUNIOR]

直接依存関係とは、プロジェクトが `go.mod` や `package.json` で明示的に宣言するパッケージです。推移的依存関係とは、直接依存関係が宣言するパッケージ、またはその推移的依存関係が宣言するパッケージです。大規模なプロジェクトでは、推移的依存関係のグラフが直接依存関係リストより 10〜50 倍大きくなる場合があります。

推移的依存関係は、プロジェクトのプロセスで実行されるコードですが、プロジェクトが選択せず、レビューもせず、多くの場合存在さえ知らないコードです。推移的依存関係における脆弱性はプロジェクトの脆弱性です: プロジェクトはそのコードを本番にデプロイしています。サプライチェーン攻撃（メンテナーのアカウントが侵害される、廃止後にパッケージがハイジャックされる、依存関係が静かに改変される）は、直接依存関係よりも見えにくい推移的パッケージを標的にすることが多いです。

推移的グラフには 2 種類のリスクが存在します:

1. **既知の脆弱性（CVE）。** 公開された共通脆弱性識別子（CVE）エントリが、特定のパッケージの特定のバージョン範囲を悪用可能と識別します。監査ツールはインストール済みのバージョングラフを CVE データベースと照合して一致を報告します。

2. **未知の脆弱性。** CVE はまだ存在しませんが、パッケージにはリスクを高める性質があります: メンテナンスされていない、シングルメンテナー、異常なポストインストールスクリプト、予期しないアクセス権の要求。これらはツールではなくポリシーと人間の判断が必要です。

### 慣用的な応用  [MID]

Meridian は CI で 2 つの監査ツールをスタックごとに 1 つずつ実行しています:

**Go 側 — `govulncheck`:**

```yaml
# .github/workflows/ci.yml (security job)
- name: Run govulncheck
  run: govulncheck ./...
```

`govulncheck`（`golang.org/x/vuln` より）は完全なモジュールグラフを Go 脆弱性データベースと照合し、バイナリで実際に呼び出されるコードから到達可能な脆弱性のみを報告します。素朴な CVE データベースとの照合と異なり、`govulncheck` はデータフロー解析を実行して、インポートされているが到達可能なコードパスで実行されることのないパッケージからの偽陽性を排除します。これは実務上重要です: Meridian が決して呼び出さないデータベースドライバのサブパッケージにある脆弱性は指摘を生成しません。

**npm 側 — `npm audit`:**

```yaml
# .github/workflows/ci.yml (security job)
- name: Run npm audit
  run: pnpm audit --audit-level=high
```

`--audit-level=high` フラグは低および中程度の深刻度のアドバイザリをフィルタリングします。これは意図的なポリシー選択です: フロントエンドの推移的依存関係における低深刻度のアドバイザリは一般的であり、多くの場合理論的なもの（開発者のマシンでローカルコードを実行する必要がある攻撃）であり、チームがすべての監査出力を無視するようになるノイズを生成します。Meridian は中程度および低いアドバイザリを月次でレビューするバックログアイテムとして扱い、CI ゲートとしてではありません。高および重大なアドバイザリはビルドをブロックします。

中程度および低いアドバイザリの月次レビューはカレンダーゲートのタスクとして実行されます: devops-engineer が監査レポートを作成し、各指摘を Meridian の攻撃面（B2B SaaS であり、パブリックな npm パッケージではない; フロントエンドはサーバーではなくユーザーのブラウザで実行される）に対して評価し、各アドバイザリを解決、受容、またはエスカレートします。

### トレードオフと制約  [SENIOR]

`--audit-level=high` フィルタは、中程度および低い脆弱性が CI をブロックせずに存在することを受け入れます。Meridian はこれを 2 つの文書化された理由で受け入れます: フロントエンドの推移的グラフにおける中程度のアドバイザリの多くは（Vite プラグイン、esbuild ラッパー）、クラウドにデプロイされた B2B SaaS のスコープにない開発者環境の攻撃シナリオを説明しています; そして、すべての中程度のアドバイザリでブロックすることは、製品変更とは無関係な継続的な手動ロックファイル介入を必要とすることで CI ゲートへの信頼を損ないます。受け入れられたポスチャーは `docs/en/adr/` に記録され、security-reviewer による四半期レビューを受けます。

- 月次の中程度/低アドバイザリレビューを統制するトリアージマトリクス (秘密管理ポリシーの文脈) については [security-mindset.md: Secrets Handling](./security-mindset.md#secrets-handling-vault-in-prod-env-example-in-repo-log-redaction-always) を参照してください。

## 廃止対応: 置き換えられたパッケージからの移行  [MID]

### 第一原理からの説明  [JUNIOR]

依存関係は、作者がメンテナンスを停止した場合、公式の後継を発表した場合、またはプロジェクトが吸収できないスケジュールで後方互換性を破るバージョンを公開した場合に廃止されます。廃止は即時ではありません: パッケージは公式に廃止されていても、本番で数ヶ月または数年間正常に機能し続ける場合があります。リスクは時間とともに蓄積されます — セキュリティ脆弱性がパッチされず、Go モジュールの互換性保証が侵食され、新しい言語バージョンとの非互換性が現れます。

廃止への対応は移行です: 廃止されたパッケージをその後継または標準ライブラリの同等品に置き換え、すべての呼び出しサイトを更新し、テストを再実行し、古いインポートを削除します。移行コストは廃止されたパッケージがアプリケーションコードとどれだけ密結合しているかによって決まります。狭いインターフェースを通じて 1 つのファイルで使用されているパッケージは午後には移行できます。型がコードベース全体にまたがるパッケージは数週間かかります。

移行時の結合の深さは、ほぼ常にパッケージが最初に追加された時点での決定によって決まります。薄いラッパーがパッケージの型をアプリケーションから隔離します; ラッパーを削除するには 1 つのファイルを変更するだけです。パッケージの型がドメイン型、サービスメソッドのシグネチャ、リポジトリインターフェースに現れている場合は、それらのサイトすべてを変更する必要があります。

### 慣用的な応用  [MID]

Meridian はこれを直接経験しました。サービスの最初のバージョンで使用していた HTTP ルータである `github.com/gorilla/mux` が 2022 年 12 月にメンテナーによってアーカイブされました。移行決定時点（アーカイブから約 8 ヶ月後）、`gorilla/mux` はまだ機能していましたが、Meridian が使用していたバージョンに影響する 2 つの CVE に対してセキュリティパッチが適用されていませんでした。

移行先は Gin（`github.com/gin-gonic/gin`）で、Meridian がミドルウェアエコシステムとパフォーマンス特性を評価していました。移行前の結合評価:

- `gorilla/mux` の型は `cmd/server/main.go`（ルータ設定）と `internal/handler/*.go`（`http.ResponseWriter` と `*http.Request` を受け付けるハンドラのシグネチャ）のみに現れていました。
- ドメイン型やサービスインターフェースは `gorilla/mux` の型を直接参照していませんでした。ハンドラ関数は `mux` 固有の型ではなく、標準ライブラリの `http.ResponseWriter` と `*http.Request` を受け付けていました。

この結合プロファイルは、移行がハンドラレイヤーの変更であることを意味しました: `main.go` のルータ初期化を入れ替え、ハンドラのシグネチャを `*gin.Context` を受け付けるよう更新し、パラメータ抽出の呼び出しを更新します（`mux.Vars(r)["id"]` から `c.Param("id")` へ）。サービスレイヤー、リポジトリレイヤー、ドメイン型は変更なしでした。

移行はエンドツーエンドで 2 営業日かかりました: ハンドラの書き直しに 1 日、CI の安定化に 1 日（完全な `gorilla/mux` ルータを構築していた 1 つのインテグレーションテストを `gin.New()` を使うよう更新する必要がありました）。ハンドラのシグネチャに `gorilla/mux` の型ではなく標準ライブラリのインターフェースを使っていた浅い結合が、移行コストを決定する要因でした。

Meridian が持ち帰った教訓: 新しい依存関係を評価する際、パッケージの型がレイヤーをまたぐシグネチャや型に現れるかを評価する。もしそうなら、アプリケーション向けのインターフェースがプロジェクト自身の型を使う薄いアダプターでパッケージをラップする。アダプターが変更される; 呼び出しサイトは変更されない。

### トレードオフと制約  [SENIOR]

`gorilla/mux` の移行を扱いやすくした浅い結合は、意図的なアーキテクチャ選択ではありませんでした。それは慣用的な Go のアプローチ — 標準ライブラリの `http.Handler` シグネチャ — であり、これが自然な隔離レイヤーを生み出します。`gorilla/mux` 固有の型をドメイン構造体やサービスメソッドのシグネチャに格納していたチームは、一桁高い移行コストを支払いました。

Gin 自体は現在、シングルメンテナーの集中リスクを抱えています。判断ツリーの基準（「直近 12 ヶ月でアクティブにリリースしているメンテナーが少なくとも 1 人いる」）は現在満たされていますが、Gin が今日アーカイブされれば、`*gin.Context` を使うハンドラのシグネチャはすべてのハンドラファイルの変更を必要とします。サービスとドメインレイヤーは再び影響を受けないでしょう。新しい移行評価をトリガーする基準: 18 ヶ月リリースなし、または Meridian のルーティングコードパスに直接影響する CVE。

### 関連節

- このハンドラのシグネチャにある標準ライブラリのインターフェース型が、この移行を扱いやすくした浅い結合を生み出した理由については [ecosystem-fluency.md: Go Interface Design](./ecosystem-fluency.md#go-interface-design) を参照してください。

## これまでの理解: すべての Dependabot 更新を自動マージ  [MID]

### これまでの理解 (改訂 2025-11-14)

Meridian の元の Dependabot 設定は、CI がパスした場合、パッチ、マイナー、メジャーを含むすべての依存関係更新 PR の自動マージを有効にしていました:

```yaml
# .github/dependabot.yml — original（誤り）
version: 2
updates:
  - package-ecosystem: "gomod"
    directory: "/"
    schedule:
      interval: "weekly"
    auto-merge: true   # ← すべての更新タイプに適用
  - package-ecosystem: "npm"
    directory: "/frontend"
    schedule:
      interval: "weekly"
    auto-merge: true   # ← すべての更新タイプに適用
```

この設定は、CI が徹底していれば自動マージは安全であると説明する Dependabot ドキュメントを読んだ後に適用されました。2 つのインシデントにより、この前提が誤りであることが明らかになりました:

1. `@tanstack/react-query` を `5.17.x` から `5.28.0` へのマイナー更新に、`staleTime` のデフォルト値への挙動変更が含まれていました。既存のテストが stale なデータのキャッシュ動作をカバーしていなかったため CI はパスしました。この変更は本番に達し、データが期待より積極的に再フェッチされてタスクリストビューで目に見えるローディングちらつきが発生するキャッシュ関連のリグレッションを引き起こしました。根本原因はマージから 2 日後に特定されました。

2. Go モジュール（`github.com/redis/go-redis/v9`）のマイナー更新で、Redis 接続が拒否された場合に返すエラー型が、ラップされた `net.OpError` からカスタムクライアントエラー型に変更されました。Meridian の `repository/idempotency.go` のエラー変換レイヤーは `net.OpError` を具体的にチェックしていました。このチェックは更新後に静かに機能しなくなり、Redis の接続失敗が期待されるサーキットブレーカーの動作ではなく 500 エラーとして現れ始めました。このリグレッションは CI ではなくアラートによって検出されました。

**修正後の理解:**

パッチ更新は CI がパスした後に自動マージします。マイナー更新は PR を開き、マージ前に人間のレビューを必要とします。メジャー更新は決して自動マージされません; 明示的な移行評価と、テストカバレッジの拡張を含む専用の PR が必要です。

修正後の Dependabot 設定:

```yaml
# .github/dependabot.yml — corrected
version: 2
updates:
  - package-ecosystem: "gomod"
    directory: "/"
    schedule:
      interval: "weekly"
    groups:
      go-patch-updates:
        update-types: ["patch"]
    # マイナーとメジャーの更新は自動マージなしで PR を開く; マージ前にレビュー。
  - package-ecosystem: "npm"
    directory: "/frontend"
    schedule:
      interval: "weekly"
    groups:
      npm-patch-updates:
        update-types: ["patch"]
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "monthly"
    # すべての Actions 更新は SHA; 月次でグループ化、マージ前にレビュー。
```

パッチのみの自動マージは、semver のパッチバージョンが契約上バグフィックスのみであるため正当化されます — パッチリリースでの挙動変更は semver の違反であり上流メンテナーの責任です。Semver 規約上、マイナーバージョンは新しい挙動を追加する場合があります; その新しい挙動は CI がカバーしない方法でアプリケーションコードと相互作用する場合があります。人間のレビューがマージ前にチェンジログの挙動変更ノートを捕捉します。

改訂されたポリシーでは要件も導入しました: Dependabot のマイナー更新 PR はすべて、CI 結果だけでなくパッケージのチェンジログを開いてレビューしなければなりません。チェンジログが文書化しているがテストがカバーしていない挙動変更を含む CI グリーンのマイナー更新は、マージ前のテスト追加の候補です。

- 完全な Dependabot YAML とマイナー更新に伴う PR レビューチェックリストについては [release-and-deployment.md: Dependabot Configuration](./release-and-deployment.md#dependabot-configuration) を参照してください。

## コーチイラストレーション（default vs. hints）

> **説明のみ。** 以下は、このドメインのタスクに対して 2 つのコーチングスタイルがどのように異なるかを示す作業事例です。ライブエージェントのコントラクトの一部ではありません。実際のエージェントの動作は `.claude/skills/learn/coach-styles/` のスタイルファイルによって規定されます。

**シナリオ:** 学習者が Meridian の Gin ルータにサードパーティの Go レート制限ミドルウェアパッケージ（代替として `golang.org/x/time/rate` も検討中）を追加しようとしています。どのパッケージを使うかをエージェントに尋ねます。

**`default` スタイル** — エージェントは「依存関係を追加する」判断ツリーに対して決定を評価します。`golang.org/x/time/rate` が Go 拡張標準ライブラリ（Go チームがメンテナンスし、独立したリリースリスクなし）であり、トークンバケットリミッターを提供することを指摘します。次に `github.com/ullete/limiter` のようなサードパーティのミドルウェアと対比させます。このライブラリは HTTP ミドルウェアラッパーと Redis バックの分散レート制限を提供します。エージェントは 2 つのオプションを検討します: Redis バックの状態を Pod 間で共有した IP ごとのレート制限が必要なら、サードパーティパッケージが真に非自明な協調問題を解決します; プロセス内のインメモリ制限で十分なら、`x/time/rate` にプロジェクト内で書いた薄いミドルウェアラッパーが正しい答えです。エージェントはどちらのスコープが必要かを尋ね、選択されたパスを実装します。`## Learning:` トレーラーが適用された判断ツリーと標準ライブラリ拡張 vs. サードパーティのトレードオフを説明します。

**`hints` スタイル** — エージェントは意思決定フレームワーク（「依存関係を追加する判断ツリー」）に名前を付け、次のメッセージを出力します:

```
## Coach: hint
Step: Evaluate golang.org/x/time/rate vs. a third-party middleware against the dependency
      decision tree before choosing.
Pattern: Dependency decision tree — stdlib wins unless non-trivial implementation is needed.
Rationale: x/time/rate covers single-process limiting without an external dependency.
           If Redis-backed distributed limiting is required across Kubernetes pods,
           that need justifies a third-party package; otherwise, it does not.
```

`<!-- coach:hints stop -->`

学習者がスコープ要件を評価し、選択した方向性を伝えます。次のターンでエージェントは意思決定フレームワークを再説明することなく選択されたパスを実装します。
