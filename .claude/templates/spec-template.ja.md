# 機能仕様書テンプレート

> 英語版: [spec-template.md](./spec-template.md)(原文・Source of Truth)

## このテンプレートの使い方

1. 仕様書を置く場所を決めます。テンプレートは場所を強制しません。よくある例:
   - 単一言語プロジェクト: リポジトリ直下の `specs/task-management.md`
   - 二言語プロジェクト: `specs/en/task-management.md` と `specs/ja/task-management.md`
   - docs 配下: `docs/specs/task-management.md`
2. このファイルを選んだディレクトリにコピーし、`kebab-case-feature-name.md` にリネームします。
3. 以下のセクションを埋め、「このテンプレートの使い方」ブロックはコミット前に削除してください。
4. 仕様書は設計書ではなく決定の記録です。ほとんどの機能で 1〜3 ページを目安に簡潔に書きます。

## 課題

解決したいユーザー側の課題は何か。誰が抱えているか。どのくらいの頻度か。現状どのように回避しているか。短い段落 1 つで。解決策に飛びつかないこと。

## ゴール

- この機能が達成すべき、具体的で検証可能な成果
- 3〜5 個を目安に。7 個を超える場合は機能を分割する

## 非ゴール

- この機能が意図的にやらないこと。スコープクリープを防ぐ
- 「忘れていた」を「そう決めた」に変える

## 受け入れ基準

リリース前に成立していなければならない、具体的で検証可能な条件。Given / When / Then 形式を推奨:

- **Given** [状態] **when** [行動] **then** [観測可能な結果]
- ...

## 主要なインタラクション

主要フローの高レベルな記述。完全なワイヤフレームではない。既存のデザインがあればリンクする。

## メトリクス

- **先行指標:** 実際の成果を測る前に、この機能が効いているか判定する指標
- **遅行指標:** 本当に気にしている成果(リテンション、売上、完了率など)

## リスクと未解決の問い

- 既知の未知。この仕様を無効化しうるものは何か
- 実装時に意図的に先送りしている決定

## スコープ外

- この反復で扱わないと明示する関連項目
